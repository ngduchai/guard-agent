I want you to make this MPI application resilient against mid-execution
process failures.

App-specific context (read carefully — this app is structurally different
from a typical C/C++ MPI workload):

  - The application is the **Sandia Structural Simulation Toolkit (SST)**,
    a parallel discrete-event simulator.  The sst-core engine is a
    separately installed C++ binary at `$HOME/.local/sst/bin/sst`.  Treat
    it as a black-box framework — the way you would treat MPI itself.  Do
    not try to modify or rebuild sst-core.
  - The "computation" you can edit consists of just two files:
      * `bench.py`   — Python configuration that builds a 16-component
                       ring on `simpleElementExample.basicLinks`
                       (~120s sim wall at 4 MPI ranks).
      * `run_sst.sh` — per-rank launcher invoked once per rank by mpirun;
                       currently `"$SST_BIN" "$script_dir/bench.py" "$@"`.
    Do NOT introduce a separate C/C++ source tree to call VeloC's C API
    directly — there is no compiler step in this app's build that would
    pick it up.

How to add resilience here:

  - sst-core ships a built-in checkpoint mechanism.  Enabling it requires
    only command-line flags on the `sst` invocation; it produces per-rank
    `.bin` checkpoint files plus a global `.sstcpt` registry under a
    checkpoint prefix directory.  Restart is performed by passing the
    latest `.sstcpt` registry path as the only positional argument to
    sst, instead of `bench.py`.
  - Modify `run_sst.sh` so that:
      1. On a fresh run (no checkpoint registry present yet), launch sst
         with `--checkpoint-sim-period=<period>` and
         `--checkpoint-prefix=<dir>` so checkpoint files are written
         periodically during execution.  A period of `4ms` (sim time) is
         appropriate for this workload (~18ms simulated time → 4-5
         checkpoints over the run).
      2. On a subsequent run (registry already exists from a prior
         attempt), launch sst with that registry's `.sstcpt` path as the
         only argument; sst will detect the registry and resume from the
         latest checkpoint.
      3. Continue to emit the existing `SST_VALIDATION=PASSED` (or
         `=FAILED`) marker on rank 0 so the validation comparator can
         match output exactly as before.
  - Place the checkpoint prefix directory under one of the VeloC
    scratch / persistent paths declared in `veloc.cfg` (you will need to
    create that config file).  This way the validation framework's
    checkpoint-observed strategy can poll those directories and detect
    the freshly-written `.bin` / `.sstcpt` files appearing during the
    run.

Why use sst-core's native checkpoint here rather than calling VeloC's C
API:

  - VeloC's API is C/C++ and would require a compiled extension; this
    app's build system does not compile any source.
  - sst-core's native checkpoint produces real on-disk checkpoint files
    in the directories the validation framework polls; that directly
    satisfies the verdict ("checkpoint observed during execution AND
    recovery output correctness AND total kill+recovery wall-time
    < 1.2x failure-free baseline").
  - Both produce the same observable behavior the validator measures
    (checkpoint files appearing during the run; recovery completes in
    less time than running the workload from scratch).
