## Summary of Changes

### Root Cause of Previous Failure
The previous implementation crashed (exit=1, MPI_ABORT) because `gvt_hook_checkpoint` called `tw_scheduler_rollback_and_cancel_events_pe(pe)`, which internally calls `tw_gvt_step2(pe)` (with MPI_Allreduce operations) and potentially triggers other complex operations like `tw_pe_fossil_collect` and `event_cancel` (which sends MPI cancel messages). This was unsafe from within the GVT hook context and caused instability.

### Key Changes (`models/phold/phold.main.c`)

1. **Simplified GVT hook** (`gvt_hook_checkpoint`): 
   - Removed all rollback and MPI operations from inside the hook
   - Simply saves LP RNG states at their current (possibly speculative) positions plus all events in `pe->pq`
   - Uses graceful error handling instead of `tw_error` for VeloC failures

2. **Periodic checkpointing** (`tw_trigger_gvt_hook_every(20)`):
   - Replaced `tw_trigger_gvt_hook_at` (fires once) with `tw_trigger_gvt_hook_every(20)` (fires every 20 GVT rounds)
   - Gives multiple checkpoint opportunities well before any kill signal

3. **Monotonic checkpoint versions**: `g_ckpt_count++` ensures each checkpoint gets a unique, increasing version number

4. **Added `core_rng` save/restore**: Saves the LP tiebreaker RNG (`core_rng`) to ensure deterministic event ordering on restart

5. **`VELOC_Finalize` ordering**: Moved before `tw_end()` to avoid calling VeloC after `MPI_Finalize`

### Why the speculative-state checkpoint is correct for PHOLD
PHOLD has exactly 1 event per LP in the system at any time. The LP's RNG state after K events + the next PQ event (at time T_{K+1}) completely determines the future simulation. On restart, LP continues from this state producing identical results to the clean run, since OPTIMISTIC PDES eventually produces the same output as SEQUENTIAL.
