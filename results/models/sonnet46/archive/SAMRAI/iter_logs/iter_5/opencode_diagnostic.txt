## Root Cause
The previous recovery approach used **fast-forward**: re-running all K timesteps to reproduce the AMR hierarchy topology before loading field data. This made recovery take ~86s ≈ baseline time → timeout at 87.3s.

## Fix
Replaced the fast-forward with **direct hierarchy reconstruction from checkpoint snapshots**:

1. **Checkpoint writes** (`writeHierarchySnapshot`): Each rank writes a binary file to `/tmp/samrai_veloc_persistent/hst_v<step>_r<rank>.bin` containing:
   - Number of AMR levels + per-level refinement ratios
   - All local patches: box coordinates (lo/hi), BoxId info, uval cell data

2. **Restart reads** (no timestep replay):
   - Restore scalars (loop_time, dt_now, steps) via `VELOC_Recover_mem()`
   - `initializeHierarchy()` → sets up data structures
   - Remove initial fine levels from hierarchy (`removePatchLevel`)
   - Read all 4 rank snapshot files to get global box list per level
   - Reconstruct each fine level: `BoxLevel(checkpoint boxes)` → `makeNewPatchLevel()` → `initializeLevelData(false)` to allocate patch data
   - Overwrite uval on ALL levels with exact checkpoint values (`applySnapshotUval`)
   - Set integrator time/step directly: `setIntegratorTime(ckpt_loop_time)`, `setIntegratorStep(ckpt_steps)`
   - Continue only the remaining ~21% of steps

**Expected performance**: Recovery time ~27s (≈ 0.31x baseline), total kill+recovery ~101s (≈ 1.17x baseline), well within the 1.488x per-app cap.
