## Root Cause (iter_1 failure)

The recovery run crashed with **SIGABRT (exit=134)** at `__fread_chk` in `readUvalFromFile`. The crash happened because:

1. `VELOC_Route_file(original, routed)` called inside `VELOC_Restart_begin` returned a path that did **not** point to the uval binary file written during the checkpoint run. It likely returned the scratch-directory path which either didn't exist or contained VeloC's internal metadata (not the binary patch data).

2. When `readUvalFromFile` opened that wrong file and read the first few bytes as `int ndim`, it got a garbage value (e.g., `ndim = 1000`). Then `fread(lo, sizeof(int), ndim, fp)` tried to read 4000 bytes into `int lo[3]` (12 bytes) ‚Üí FORTIFY buffer overflow ‚Üí `__fread_chk` assertion ‚Üí **SIGABRT**.

## Fix Applied

In `main.cpp`:

1. **Write uval to a predictable absolute path** (`/tmp/samrai_veloc_persistent/uval_v{VERSION}_r{RANK}.bin`) BEFORE the VeloC checkpoint. This path is fully controlled and doesn't depend on `VELOC_Route_file` behavior.

2. **On restart, read from the predictable absolute path** using `restart_version` (from `VELOC_Restart_test`). No `VELOC_Route_file` call during restart for reading.

3. **Added `ndim` bounds check** (`if (ndim < 1 || ndim > 3) break`) in `readUvalFromFile` as defensive programming against wrong files.

4. Still call `VELOC_Route_file` during checkpoint (write phase) to also give VeloC a copy of the file - this ensures VeloC counts more checkpoint files.
