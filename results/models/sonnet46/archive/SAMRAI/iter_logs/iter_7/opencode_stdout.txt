Here is a summary of the diagnosis and fix:

## Root Cause Analysis

The previous iteration (iter 6) had checkpoint files being correctly written and detected by the validator (3472 files, 876MB), but the recovery produced wrong output (`max_abs_diff=4.92e+08`).

**The bug**: The old code wrote level-0 snapshot files directly to `/tmp/samrai_veloc_persistent/l0snap_v000040_r0000.bin` — outside of VeloC's file management. When VeloC ran `VELOC_Checkpoint_end(1)` in sync mode, it **cleared the persistent directory and replaced its contents** with only the files it managed. This deleted our snapshot files, causing `applyLevel0Snapshot` to silently fail (file not found), and the simulation ran the remaining steps with wrong (t=0) initial data.

## Fix

The core change is writing the level-0 snapshot **within** VeloC's checkpoint lifecycle using `VELOC_Route_file`:

1. **During checkpoint**: `VELOC_Route_file("l0snap_r%04d.bin", routed)` is called between `VELOC_Checkpoint_begin` and `VELOC_Checkpoint_end`. VeloC now owns the file and persists it correctly.

2. **During restart**: The snapshot is read via `VELOC_Route_file` between `VELOC_Restart_begin` and `VELOC_Restart_end`. VeloC provides the path to the correct checkpoint version's file.

3. **Buffering**: The snapshot data is read into memory **before** `initializeHierarchy()` (which would reset level-0 data), then applied to the fresh hierarchy's level-0 patches afterward.

4. **Cleanup**: Removed the now-unnecessary separate `applyLevel0Snapshot` function (dead code from old approach).

The restart flow is: restore scalars → read snapshot to memory buffer → initialize hierarchy → apply buffer to level-0 patches → remove fine levels → reset hierarchy config → set integrator time/step → resume remaining timesteps.
