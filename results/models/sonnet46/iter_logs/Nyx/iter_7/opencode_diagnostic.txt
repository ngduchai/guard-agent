Here is a summary of what I changed and why:

## Summary of Changes

### Root Cause Analysis (iter_6 failures)

1. **Recovery timeout**: The simulation ran from scratch (t=0) during recovery, taking the full ~110s and timing out at 108.5s. The COMPLETE marker + sentinel mechanism in iter_6 was unreliable - likely the COMPLETE marker was being checked but failing for edge cases.

2. **Wrong output (iter_5)**: The fine level (level 1, 2) AMR grids had wrong BoxArrays after restart. The checkpoint BoxArrays differ from the fresh-init BoxArrays because the Sod shock moves during the simulation.

### Fix: Complete rewrite of `nyx_main.cpp`

**Key improvements**:

1. **Reliable checkpoint detection** (`findLatestCompleteCheckpointStep`): Directory scan using `dirent.h` to find the highest `step_N` directory with a `DONE` marker. This is simple, robust, and doesn't depend on VeloC's internal state.

2. **Correct write ordering**: AMR data is written BEFORE the VeloC commit (`VELOC_Checkpoint_begin/mem/end`). The DONE marker is written after all AMR data is on disk. This ensures: if VeloC says version N is available → DONE for N exists → AMR data for N is complete.

3. **BoxArray persistence**: The header file now stores the `BoxArray` for each AMR level. This allows exact restoration of fine level grid structures.

4. **Proper fine level restoration**: Instead of using `amrptr->regrid()` (which was causing issues in iter_6), the new code:
   - Destroys all existing fine levels
   - Creates new Nyx levels with the checkpoint BoxArrays (via `getLevelBld()`)
   - Fills MultiFabs directly with checkpoint data via `VisMF::Read` + `ParallelCopy`
   - Calls `post_regrid` to finalize

5. **No regrid during restart**: Eliminates the expensive and fragile `amrptr->regrid(0, ckpt_cumtime)` call that was causing timeouts.
