## Summary of Changes

**File modified:** `Source/Driver/nyx_main.cpp`

**Root cause of iter_4 failure:** The recovery (attempt 2) ran for the full 139.2s timeout and exited with code 1. This happened because `VELOC_Restart_test("nyx", INT_MAX)` returned `VELOC_FAILURE` after the previous process was SIGKILL'd. When `readVelocCheckpoint` returned `false`, the simulation started from scratch (step 0), took the full 110s to run, and was killed by the 139.2s timeout without producing `validation_output.bin`.

**Fix:**
1. Added `NYX_CKPT_SENTINEL = "/tmp/nyx_veloc_persistent/nyx_ckpt_latest_step.txt"` - an own sentinel file in the VeloC persistent directory (which is NOT cleared between attempt 1 and attempt 2).

2. In `writeVelocCheckpoint`: After all AMR data is written AND `VELOC_Checkpoint_end()` is called, rank 0 writes a `COMPLETE` marker file in the checkpoint directory, then atomically updates the persistent sentinel file with the step number using `rename()`.

3. In `readVelocCheckpoint`: Instead of relying on `VELOC_Restart_test` (which fails after SIGKILL), read the sentinel file to get the latest complete step number, verify the `COMPLETE` marker exists, and restore from that checkpoint directory. The VeloC restart API calls are removed from the recovery path to avoid potential hangs.

**Why this works:**
- The validator clears VeloC scratch/persistent dirs before attempt 1 (not before attempt 2)
- Attempt 1 creates checkpoints, writes `COMPLETE` markers, and updates the sentinel
- When attempt 1 is SIGKILL'd, the sentinel points to the last FULLY written checkpoint
- Attempt 2 starts, finds the sentinel, verifies the `COMPLETE` marker, and correctly restores from that checkpoint
- Recovery runs from the restored checkpoint state to completion, producing `validation_output.bin`
