## Summary

The previous attempt (iter_4) failed with `output_correct=False` after recovery (`max_abs_diff=2.236e-01`). The iter_4 fix tried to solve a pointer aliasing bug (from `cycleSolutionArrays()` swapping internal Sarray data pointers) by calling `VELOC_Mem_unprotect`/`VELOC_Mem_protect` before each checkpoint, but this approach didn't work correctly with VeloC.

**Root cause analysis:**
- `cycleSolutionArrays()` uses `Sarray::reference()` to swap internal data pointers among U, Um, Up arrays
- After N cycles, `U[g].c_ptr()` may point to a different physical buffer than was originally registered with VeloC
- Attempting to fix this by unregistering/re-registering with VeloC before each checkpoint introduced a different problem (VeloC's behavior with re-registration was not reliable)

**Fix applied:**
- Allocate **fixed checkpoint buffers** (`ckpt_U[g]`, `ckpt_Um[g]`) that never change address
- Register these fixed buffers with VeloC ONCE at initialization
- Before each `VELOC_Checkpoint`: **explicitly memcpy** the current U[g]/Um[g] data into the fixed buffers
- After `VELOC_Restart`: **explicitly memcpy** the restored data from fixed buffers back into U[g]/Um[g]

This approach is robust regardless of any pointer aliasing from `cycleSolutionArrays()` and regardless of any VeloC quirks with re-registration. The recovery should produce bit-identical results to a clean run since the checkpoint captures the exact floating-point state at step N, and continuation from step N+1 gives the same result as a full run.
