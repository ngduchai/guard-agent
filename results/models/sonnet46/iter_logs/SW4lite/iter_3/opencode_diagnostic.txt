Here's a summary of what was wrong and what was fixed:

## Root Cause
The previous attempt failed because checkpoint data consumed 11+ GB (exceeding the 10.7 GB disk-guard limit), causing the application to be killed by the validator before completing.

Two issues caused this:
1. **Wrong config key**: `max_ckpts = 2` is not a valid VeloC key. The correct keys are `max_versions` (for persistent) and `scratch_versions` (for scratch). Since the key was ignored, VeloC kept ALL checkpoint versions indefinitely.
2. **Checkpoint too frequently**: Every 10 timesteps → ~10+ versions × ~1.1 GB = 11+ GB total.

## Fixes Applied

### `veloc.cfg`
- Changed `max_ckpts = 2` → `max_versions = 1` + `scratch_versions = 1` (correct VeloC retention config keys)

### `src/EW.C`
- Added `#include <dirent.h>` for directory scanning
- Added `pruneVelocDir()` / `pruneVelocCheckpoints()` helper functions that scan scratch/persistent dirs and delete checkpoint files older than a given version threshold (handles both flat and subdirectory VeloC layouts)
- Changed checkpoint frequency from every 10 steps → every `mNumberOfTimeSteps/4` steps (~4 checkpoints per run, keeping at most 2 versions on disk at any time)
- After each `VELOC_Checkpoint()` call, manually prune files from both `/tmp/sw4lite_veloc_scratch` and `/tmp/sw4lite_veloc_persistent` that are older than the previous checkpoint version

The peak disk usage will now be at most 2 checkpoint versions × ~1.1 GB/version = ~2.2 GB, well within the 10.7 GB limit.
