Here's a summary of what was changed and why:

## Root Cause (iter 4 failure)

The iter 4 code used `VELOC_Route_file()` to manage field data files separately from VeloC's memory protection. During restart, `VELOC_Route_file()` was failing to locate the field data files (returning `VELOC_FAILURE`). This triggered the error path that **explicitly reset `istep[0] = 0`**, causing the simulation to re-run all 80,000 steps from the beginning. The ratio was 1.007 (essentially a full re-run), explaining `recovery_actually_resumed=False`.

## Fix Applied

**`Source/Evolve/WarpXEvolve.cpp`** - Complete redesign of the VeloC checkpoint/restart logic:

1. **Eliminated `VELOC_Route_file()`** entirely - no more manually-managed field data files that could fail during routing.

2. **Pre-allocated field buffer** (`static std::vector<char> field_buf`): At initialization time, compute the total bytes needed for all local E/B field FABs, allocate a buffer, and register it with `VELOC_Mem_protect(5, field_buf.data(), fsize, 1)`. This makes VeloC manage the field data buffer using the same reliable mechanism as scalars.

3. **Checkpoint**: Before each `VELOC_Checkpoint_mem()` call, pack all field FAB data into `field_buf`. VeloC then saves both scalars (IDs 1-4) and the field buffer (ID 5) atomically.

4. **Restart**: After `VELOC_Recover_mem()`, both scalars AND `field_buf` are correctly restored. Then unpack from `field_buf` into the live MultiFabs.

5. **No fallback reset**: If restart succeeds (which it now will, since all data is in VeloC mem protect), `istep[0]` is properly restored and the loop starts from the checkpoint step - not from step 0.
