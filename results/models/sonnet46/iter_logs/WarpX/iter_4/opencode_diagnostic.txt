## Summary of Changes

### Root Cause Analysis
The previous implementation (`iter_3`) failed with:
1. **Segfaults** (`malloc(): invalid size`) during restart due to `FArrayBox::readFrom(ifs)` corrupting the heap when reading back FAB data from files
2. **Recovery timeout** (108s > 107s limit) because the segfault prevented actual restart, causing the sim to run from scratch

### Fixes Applied to `Source/Evolve/WarpXEvolve.cpp`

1. **Replaced `FArrayBox::readFrom`/`writeOn` with raw `memcpy`**: The old API reads/writes box metadata headers which corrupted heap on restart. The new approach uses `fab.dataPtr()` + `fab.nBytes()` for direct memory copies.

2. **Single combined binary file per rank**: Instead of one VeloC-routed file per field direction (which was failing silently), all Efield_fp and Bfield_fp data is packed into a single binary blob per MPI rank. This ensures VeloC properly tracks the file.

3. **Three helper lambdas**: `collect_field_data` (count bytes), `pack_field_data` (write to buffer), `unpack_field_data` (read from buffer) — all using the same deterministic field ordering, ensuring checkpoint/restart byte alignment.

4. **Added `<cstring>` include** for `std::memcpy`.

5. **Checkpoint interval changed to 2000 steps** (≈2s of wall time) for slightly less I/O overhead.

### Expected Outcome
- 16 checkpoint files (4 ranks × 2 files × 2 versions) will be detected
- On restart: scalars restored via `VELOC_Recover_mem()`, field data read from files via `memcpy` into existing FABs
- Recovery runs ~36000 remaining steps ≈ 35s, well within the 107s timeout
- Recovery ratio ≈ (3s init + 35s sim) / 78s ≈ 0.49x < 1.2x threshold
