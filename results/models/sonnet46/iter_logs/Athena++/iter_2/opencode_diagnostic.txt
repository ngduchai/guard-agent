## Root Cause of iter_1 Failure

The build failed with "undefined reference to `VELOC_Init`" (and other VeloC symbols). Although `src/main.cpp` included `<veloc.h>` and the `Makefile` had the correct linker flags (`-L/home/ndhai/usr/lib -lveloc-client`), the validator's build command calls `configure.py` before `make`. This regenerates `Makefile` from `Makefile.in`, **overwriting** the VeloC linker flags that iter_1 added directly to `Makefile`. Since iter_1 never modified `Makefile.in`, the regenerated Makefile had no VeloC flags.

## Changes Made

### 1. `Makefile.in` — Primary fix
Added VeloC include and library flags directly into the template:
```makefile
CPPFLAGS := @PREPROCESSOR_FLAGS@ -I/home/ndhai/usr/include
LDFLAGS := @LINKER_FLAGS@ -L/home/ndhai/usr/lib -Wl,-rpath,/home/ndhai/usr/lib
LDLIBS := @LIBRARY_FLAGS@ -lveloc-client
```
Now any `configure.py` regeneration preserves the VeloC flags.

### 2. `src/main.cpp` — Updated VeloC config path
Changed `VELOC_Init` to use the **absolute path** to `veloc.cfg`:
```cpp
static const char veloc_cfg_path[] =
    "/home/ndhai/diaspora/guard-agent/build/tests_baseline_sonnet46/Athena++/veloc.cfg";
VELOC_Init(MPI_COMM_WORLD, veloc_cfg_path);
```
The validator runs the binary from a temporary output directory (not the source tree root), so a relative path `"veloc.cfg"` would fail at runtime. The absolute path ensures VeloC always finds its config.
