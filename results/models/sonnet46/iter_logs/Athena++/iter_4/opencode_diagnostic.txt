If `veloc_write_state()` fails (which might happen if the file routing fails), `state_ok = false`, then `VELOC_Checkpoint_mem()` is NOT called, and `VELOC_Checkpoint_end(0)` is called. This tells VeloC the checkpoint is INVALID. VeloC should NOT record this version. So `VELOC_Restart_test` returns `VELOC_FAILURE` because there's no valid checkpoint.

**This is the bug in iter_3!** The routed file writing might be failing (VELOC_Route_file returning an error or the file write failing), causing `veloc_write_state()` to return false, then `VELOC_Checkpoint_end(0)` is called (invalid checkpoint), and restart fails.

In my new implementation, I use ONLY the memory buffer (no file routing). The `veloc_pack_state()` should work reliably. If it fails (buffer overflow), `VELOC_Checkpoint_begin` is never called, so no invalid checkpoint is written.

If `VELOC_Checkpoint_mem()` fails (`rc_mem != VELOC_SUCCESS`), then `VELOC_Checkpoint_end(0)` is called ‚Üí invalid checkpoint. But `VELOC_Checkpoint_mem()` should not fail if `VELOC_Mem_protect` was called correctly.
