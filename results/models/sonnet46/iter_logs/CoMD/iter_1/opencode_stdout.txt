**Strategy:**

The CoMD main loop runs `nSteps` timesteps in chunks of `printRate`. On restart, I need to restore from the last checkpoint and continue from there.

The key state to checkpoint per rank:
- `iStep` - current step number (drives the loop)
- `atoms->nLocal` - number of local atoms
- `atoms->gid[0..nTotalAtoms-1]` - global atom IDs
- `atoms->iSpecies[0..nTotalAtoms-1]` - species per atom
- `atoms->r[0..nTotalAtoms-1]` - positions
- `atoms->p[0..nTotalAtoms-1]` - momenta
- `atoms->f[0..nTotalAtoms-1]` - forces
- `atoms->U[0..nTotalAtoms-1]` - potential energy per atom
- `boxes->nAtoms[0..nTotalBoxes-1]` - atoms per link cell

The approach:
1. `VELOC_Init(MPI_COMM_WORLD, "veloc.cfg")` after `initParallel`
2. After `initSimulation`, register all memory regions
3. Check `VELOC_Restart_test` to see if a prior checkpoint exists
4. If yes, restore and set `iStep` to the checkpointed step
5. In the main loop, checkpoint after each `printRate` batch
6. `VELOC_Finalize(0)` before `destroyParallel`

Important nuance: The `nTotalBoxes` is fixed by the domain decomposition and doesn't change. The atom arrays are allocated as `MAXATOMS * nTotalBoxes` elements. I need to checkpoint the full allocated arrays (since atom positions in link-cell layout depend on which box they're in), plus `nAtoms` per box, plus `nLocal`.

   - `VELOC_Restart_test` to probe for existing checkpoints; if found, `VELOC_Restart` loads all registered regions back (restoring `iStep` so the loop continues from where it left off)
   - `VELOC_Checkpoint(VELOC_CKPT_NAME, iStep)` after every print interval (every `printRate` steps)

**Key design choices:**
- The checkpoint version number is `iStep` (the step after the completed batch), which monotonically increases â€” satisfying VeloC's requirement that version numbers increase
- All atom arrays are registered at their full allocated size (`MAXATOMS * nTotalBoxes`) â€” this captures the link-cell-indexed layout faithfully
- On restart, `iStep` is restored from the checkpoint, so `for (; iStep<nSteps;)` continues correctly from the restored step
- The `VELOC_Restart_test` second argument `0` means "find latest version < 0" which in VeloC semantics means "find the latest available" (version 0 = no upper bound constraint)
