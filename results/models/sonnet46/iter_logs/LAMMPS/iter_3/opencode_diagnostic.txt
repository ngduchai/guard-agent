The implementation is complete. Let me provide a summary of the changes made:

## Summary of Changes

### Root Cause of Previous Failure
The previous iteration's checkpoint/restart failed because:
1. **Stale pointer bug**: `VELOC_Mem_protect` was called once at the start of `run()` with pointers to atom arrays (`atom->x[0]`, `atom->v[0]`, etc.). During the run, `atom->avec->grow()` may reallocate these arrays, making the registered VeloC pointers stale. At checkpoint time, VeloC saved data from old (freed) memory, producing garbage.

2. The output after restart had `max_abs_diff=8.041e+02` — a huge error indicating wrong checkpoint state was restored.

### Solution in `src/verlet.cpp`

Replaced the `VELOC_Mem_protect`-based approach with a **file-based VeloC API** that explicitly writes current atom data at each checkpoint:

**Checkpoint**: Each MPI rank:
1. Packs its current local atoms into a `VelocAtomRec` buffer (tag, x[3], v[3], image, type, mask)
2. Calls `VELOC_Checkpoint_begin` + `VELOC_Route_file` to get a per-rank file path
3. Writes: `[bigint ntimestep][int nlocal][nlocal × VelocAtomRec]`
4. Calls `VELOC_Checkpoint_end`

**Restart**: Each MPI rank:
1. Calls `VELOC_Restart_test` to find the latest checkpoint
2. Calls `VELOC_Restart_begin` + `VELOC_Route_file` to get its own file path
3. Reads and directly restores `atom->tag`, `atom->x`, `atom->v`, `atom->image`, `atom->type`, `atom->mask`, and `atom->nlocal`
4. Calls `comm->borders()` to rebuild ghost atom lists (no `comm->exchange()` — that would move atoms too early, breaking bit-perfect recovery)
5. Calls `neighbor->build(1)` to rebuild neighbor lists
6. Recomputes forces `f(T)` at the checkpoint configuration (required for `initial_integrate()` to correctly advance velocities)
7. Restores `update->ntimestep` and `atom->nextsort`

This approach preserves atom ordering exactly, enabling bit-for-bit identical trajectory recovery.
