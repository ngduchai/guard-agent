Here's a summary of what I found and fixed:

## Root Cause

The recovery crashed with "double free or corruption (out)" because `veloc_do_restart()` called `mesh->mesh_memory.memory_delete_all()` and `state->state_memory.memory_delete_all()`. These functions call `free()` on **every** entry in the MallocPlus dictionaries.

The problem: The `State` constructor registers `cpu_timers` and `gpu_timers` (which are **member arrays** of the State object, not heap-allocated) via `memory_add()`. When `memory_delete_all()` called `free()` on those member-array pointers, it corrupted the heap, causing the subsequent crash.

## Fix

Replaced the dangerous `memory_delete_all()` + `allocate()` pattern with targeted `memory_realloc(n, "name")` calls on only the individual per-cell arrays that actually need resizing (i, j, level, H, U, V). This:

1. Only calls `realloc()` on heap-allocated arrays registered via `memory_malloc()`
2. Leaves all `memory_add()`-registered entries (cpu_timers, gpu_timers, etc.) untouched
3. Correctly updates the MallocPlus pointer dictionary for subsequent termination cleanup
4. Calls `mesh->memory_reset_ptrs()` to refresh raw pointers after any realloc-induced pointer changes

The `veloc.cfg` was already correct from iteration 2 and needs no changes.
