Here's a summary of what was fixed and why:

## Root Cause Analysis

Both Test 1 (failure-prone/restart) and Test 2 (failure-free) were failing with large numeric differences (~500-800 units). The issue was in the chunked run approach.

**The bug**: When running the simulation in chunks (each chunk calls `lammps->input->one("run N")`), the LAMMPS `run` command with default `pre yes` calls `lmp->init()` and `integrate->setup(1)` on **every chunk**, not just the first. `Verlet::setup(1)` calls `neighbor->build(1)` which resets `neighbor->ago = 0`. With `neigh_modify every 20 check no`, this caused the neighbor list to be rebuilt at steps 20, 40, ... relative to the **start of each chunk** rather than relative to the **overall simulation start**. This produced a different trajectory than the original `run 33000` command.

## Fix Applied

Modified `run_veloc()` in `src/main.cpp` to use appropriate `pre`/`post` flags:

- **First chunk** (fresh start or after restart): `run N post no` — uses default `pre yes` to properly initialize LAMMPS (calls `lmp->init()`, `neighbor->build()`, etc.)
- **Middle chunks**: `run N pre no post no` — skips reinit, maintains continuous neighbor rebuild schedule
- **Last chunk**: `run N pre no` — skips reinit, allows final performance summary output

This ensures the simulation trajectory is identical to a single `run 33000` command, while still writing VeloC checkpoints after each chunk for fault tolerance.
