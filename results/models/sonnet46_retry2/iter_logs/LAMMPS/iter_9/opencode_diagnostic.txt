## Summary

**Problem in iter_8**: Recovery crashed with SIGSEGV (exit code 139) in just 2.2 seconds.

**Root Cause**: In the restart path, `lammps->init()` was called directly (without going through `Run::command()`). The critical issue: `Update::init()` checks `if (whichflag == 0) return;` at line 124 of `update.cpp`. Since `whichflag` is initialized to 0 in the `Update` constructor and `Run::command()` sets it to 1 before calling `lmp->init()`, calling `lammps->init()` directly left `whichflag=0`, causing `Update::init()` to return early. This meant:
1. `Verlet::init()` was **never called** — leaving `pair_compute_flag`, `force_compute_flag`, `triclinic`, etc. **uninitialized**
2. `first_update` was **never set to 1**
3. When `setup_minimal(1)` accessed these uninitialized fields, it crashed

**Fix**: Added `lammps->update->whichflag = 1;` immediately before `lammps->init()` in the restart path (`main.cpp:348`). This replicates what `Run::command()` does before calling `lmp->init()`, ensuring `Verlet::init()` is properly called.

**Addressing iter_7's 773 diff**: The iter_8 approach uses `setup_minimal(1)` instead of `setup(1)`. The key difference is that `setup(1)` always calls `atom->sort()`, which in iter_7 caused an extra sort at step 23100 (the checkpoint step). In the original run, the last sort was at step 23000 (`nextsort=24000>23100`), so no sort at step 23100. This extra sort in iter_7's restart changed the atom ordering, causing the numerical discrepancy. With `setup_minimal(1)`, no sort is performed, and `nextsort` is set correctly via the formula `(saved_ntimestep/sortfreq)*sortfreq + sortfreq`, matching the original run's state exactly.
