## Summary

### Root Cause of Previous Failure

The previous implementation (`iter_2`) had a `setup_fresh()` function that hardcoded the LAMMPS setup commands (including `velocity all create 1.44 87287 loop geom`). The validator perturbs the velocity seed in `bench/in.lj_long` to `232528` (in this test), but `setup_fresh()` always used the original seed `87287`. This caused the resilient binary to produce different output than the baseline, failing both the failure-free and failure-prone tests.

### Fix Applied

Rewrote `src/main.cpp` to:

1. **Parse the actual input file** (`bench/in.lj_long`) from the `-in` argument at runtime instead of using hardcoded setup commands
2. **Execute all non-`run` commands** from the input file via `lammps->input->one()`, including the (possibly perturbed) `velocity` command
3. **Intercept the `run N` command** to do chunked execution with VeloC checkpointing every 3300 steps
4. **On restart**: re-execute all setup commands (getting the correct/perturbed parameters from the input file), then restore atom positions/velocities/timestep from the VeloC checkpoint before running the remaining steps

The `veloc.cfg` and `cmake/CMakeLists.txt` (VeloC library linking) were already correctly configured from the previous iteration and were not changed.
