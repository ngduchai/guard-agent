## Summary

**Root Cause**: When restoring from a VeloC checkpoint, the code calls `execute_setup_commands()` which uses `create_atoms` to place N₀ atoms on each MPI rank. After `create_atoms`, only atoms 0..N₀-1 have valid `type` and `mask` values.

During MD simulation, local atom counts fluctuate as atoms diffuse between processor domains. At checkpoint time (step ~23100), some ranks have more local atoms than N₀. When restoring `saved_nlocal > N₀` atoms from the checkpoint, atoms at indices N₀..saved_nlocal-1 get correct positions and velocities from the checkpoint buffers, but have **uninitialized `type` and `mask` values** because `create_atoms` only initialized the first N₀ atom slots.

When `comm->exchange()` processes these atoms during setup, it sends them to neighboring ranks with garbage `type` values. The pair force computation (`pair_lj_cut`) uses `type[i]` to look up force coefficients (`lj1[itype][jtype]`, `cutsq[itype][jtype]`). Type=0 (or garbage) causes incorrect force lookup, leading to wrong forces and divergent trajectory.

**Fix Applied** (`src/main.cpp`):
1. Added `#include "atom_vec.h"` to access `AtomVec::grow()`
2. Record `nlocal_0` (initial local atom count from `create_atoms`) before allocating checkpoint buffers
3. In the restart path: if `restore_count > nmax`, grow atom arrays first; then initialize `type=1` and `mask=1` for all atoms at indices `nlocal_0..restore_count-1`

This ensures all restored atoms have valid pair-coefficient lookup indices regardless of how many more atoms a rank has at checkpoint time versus simulation start.
