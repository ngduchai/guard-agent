Here's a summary of the changes made:

## Root Cause

The previous implementation (iter 3) failed because:
1. The checkpoint stored all levels' patch data, but matched patches by `local_id`
2. After `initializeHierarchy()`, fine levels had patches in the **initial condition sphere position** (~(5.5, 5.5)), while checkpoint fine levels had patches at the **advected sphere position** (~(15, 10) at checkpoint time)
3. Since the boxes didn't match spatially, fine level checkpoint data wasn't restored
4. Fine levels were removed and later recreated via regridding with only **interpolated coarse data**
5. During 1-2 steps without fine levels, the coarse level advanced without proper AMR flux corrections, causing O(1) error that propagated to give ~52% relative error in sum2

## Fix Applied

**New checkpoint format** (magic=0x4C56434C):
- Writes **interior box corners** (lo, hi) for each patch (not local_id or ghost box)
- Writes **interior-only cell data** (no ghost cells) for efficient storage
- Includes `num_levels` in header

**New restore logic** (`restoreCheckpoint`):
1. Reads all checkpoint data into a map: `(level, lo0, lo1, hi0, hi1) → data`
2. Restores level 0 by matching interior box corners (deterministic structure)
3. Removes IC fine levels (wrong spatial position)
4. Calls `gridding_algorithm->regridAllFinerLevels(0, ...)` to recreate fine levels at the **correct** positions (from restored level 0 gradient)
5. For each newly created fine-level patch, looks up checkpoint data by box corners and restores exact values
6. Patches without a checkpoint match retain interpolated data from level 0 (fallback)

This ensures fine levels are present from the very first step of recovery, preventing the AMR flux correction error that caused the large output mismatch.
