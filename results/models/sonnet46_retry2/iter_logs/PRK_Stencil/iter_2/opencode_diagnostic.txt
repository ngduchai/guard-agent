## Summary of Fixes

### Problem Analysis (from iter_1 failures)

1. **Exit code 255 on failure-free run** (line 85-86 in validate_stdout.txt): The program crashed immediately. Root cause: the validator copies `veloc.cfg` to `resilient_clean/veloc.cfg` but the binary looked for `../veloc.cfg` (one level UP from `resilient_clean/`), which doesn't exist.

2. **Disk guard triggered** (attempts 2 and 3): The previous code wrote BOTH the `in` AND `out` arrays to VeloC, with no version limits. Each checkpoint was ~16 GB (2 arrays × ~2 GB × 4 ranks × scratch+persistent). Without `scratch_versions` or `max_versions` limits, checkpoints accumulated, blowing the 10.74 GB disk limit.

3. **`VELOC_Restart_test` with version=0**: The API looks for checkpoints with version < needed_version. Passing 0 means "find version < 0" which is always impossible. Changed to `INT_MAX` to find the latest available checkpoint.

4. **`CKPT_FREQ=5`**: Way too frequent (checkpoint every 5 iterations, ~300ms). Combined with no version limits, this created massive disk accumulation.

### Changes Made

**`veloc.cfg`**:
- Added `scratch_versions = 1`: Keep only the latest checkpoint in scratch (prevents accumulation)
- Added `max_versions = 1`: Keep only the latest checkpoint in persistent  
- Added `persistent_interval = 999999`: Skip persistent writes entirely (~11.5 days interval), so data only occupies scratch (~8.7 GB max vs 17+ GB with both scratch+persistent)

**`Stencil/stencil.c`**:
- **Dynamic veloc.cfg path**: `access("./veloc.cfg")` first (failure-free run), fallback to `../veloc.cfg` (kill-attempt runs where CWD is one level deeper)
- **Only checkpoint `out`**: Removed `VELOC_Mem_protect` for the `in` array (halves checkpoint size from ~16 GB to ~8 GB)
- **Reconstruct `in` on restart**: After K iterations, `IN(i,j) = COEFX*i + COEFY*j + (K+1)`. Ghost cells are repopulated by halo exchange in the first post-restart iteration.
- **Fixed `VELOC_Restart_test`**: Changed version=0 to INT_MAX
- **Reasonable checkpoint frequency**: Default to `iterations/20` (every 5% of run), capped to ensure checkpoint exists before the 50% kill point; respects `CKPT_EVERY` env var if within sane range
