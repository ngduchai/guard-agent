**What Changed**
- Kept the required static `veloc.cfg` valid:
  - `scratch = /tmp/smilei_veloc_scratch`
  - `persistent = /tmp/smilei_veloc_persistent`
  - `mode = sync`
- Reworked persistence to store the actual serialized Smilei state through VeloC protected memory, not a sidecar routed file.
- Added a fixed 8 MiB protected payload buffer plus a small protected marker.
- Registered both VeloC memory regions once after `VELOC_Init` and kept them registered for the full run.
- Checkpoint now:
  - serializes timestep, times, moving-window state, EM fields, and particle arrays into `smileiVelocState`
  - copies that real serialized state into the VeloC-protected payload buffer
  - stores payload size in the protected marker
  - commits both regions with `VELOC_Checkpoint_mem`
- Restart now:
  - uses `VELOC_Restart_test`
  - recovers both protected regions with `VELOC_Recover_mem`
  - reconstructs the serialized state from the recovered payload buffer
  - deserializes it back into Smilei objects

**Why**
- The previous validator rejection showed the app segfaulting at the first checkpoint and only producing a 36-byte checkpoint marker.
- The large serialized state was not being captured by VeloC checkpoint directories, so recovery had no real application state.
- This version makes VeloC persist the actual state buffer directly, while avoiding repeated protect/unprotect calls during each checkpoint.

**Verification**
- Consulted the required previous validator output first.
- Checked the installed VeloC public API in `/home/ndhai/usr/include/veloc.h`.
- Confirmed `veloc.cfg` exists with absolute, distinct `/tmp` paths.
- I could not run a build or MPI validation here because this environment does not expose a shell execution tool.
