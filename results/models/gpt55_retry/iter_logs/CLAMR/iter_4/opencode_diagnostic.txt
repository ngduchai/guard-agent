The edit now removes the restart-session reopen. I’m doing one more static pass for state that is computed in initialization but lost when `recover_checkpoint_if_available()` overwrites the mesh.
Implemented a targeted fix for the recovery crash reported in `iter_3`.

What changed:

- `clamr_mpionly.cpp`
  - Split VeloC data-region registration into `protect_checkpoint_data_regions()`.
  - Changed restart recovery from two separate `VELOC_Restart_begin()` / `VELOC_Restart_end()` sessions to one restart transaction:
    - recover metadata first,
    - resize mesh/state buffers,
    - protect resized simulation buffers,
    - recover `i`, `j`, `level`, `H`, `U`, `V`,
    - then call `VELOC_Restart_end()`.
  - Kept recovery reconstructing MPI distribution metadata from recovered per-rank cell counts instead of trusting/restoring vector storage.
  - Preserved the post-checkpoint invariant that persisted arrays contain real cells only; ghost cells and neighbor communication setup are rebuilt by `calc_neighbors_local()`.

Why this addresses the validator failure:

- The previous validation failed because the recovery run segfaulted immediately with exit code `139` and produced no `validation_output.bin`.
- The existing restart path reopened the same VeloC checkpoint after a selective metadata recovery. That is fragile with VeloC memory registration changes between restart sessions.
- The new path keeps metadata and data recovery inside one VeloC restart transaction, so VeloC does not need to close/reopen checkpoint state between recovering metadata and recovering resized buffers.

Confirmed:

- Required static `veloc.cfg` already exists and is valid:
  - `scratch = /tmp/clamr_veloc_scratch`
  - `persistent = /tmp/clamr_veloc_persistent`
  - `mode = sync`
- The prior allocator ownership changes are present and still intact.
- I could not run an MPI build/validation from this tool environment, so verification here is static source consistency against the validator failure.
