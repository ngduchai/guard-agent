What changed:

- `src/main.cpp`
  - Changed VeloC recovery flow to initialize Athena++ normally first with `pmesh->Initialize(0, pinput)`, then load VeloC-protected state into the initialized mesh.
  - This avoids using Athena++’s restart initialization mode for VeloC memory recovery, which can skip normal problem setup and alter failure-free behavior.
  - Resets the next checkpoint deadline after a successful VeloC recovery based on recovered simulation time.
  - Keeps real VeloC persistence for mesh time/cycle fields and local MeshBlock hydro conserved arrays.

- `inputs/hydro/athinput.blast`
  - Restored the validation input’s original AMR settings:
    - `refinement = adaptive`
    - `numlevel = 2`
    - `derefine_count = 5`
  - The previous attempt changed this to fixed mesh, which made the resilient clean run simulate a different problem than the perturbed vanilla baseline. That matches the iter 3 symptom: failure-prone and failure-free both exited successfully but had the same numeric mismatch.

- `veloc.cfg`
  - Confirmed the required static root config exists with absolute, distinct `/tmp` paths and `mode = sync`.
  - Existing retention limits remain in place:
    - `max_versions = 2`
    - `scratch_versions = 2`

Why this targets the validator failure:

- Iter 3 failed because output correctness failed for both recovery and failure-free runs, not because checkpoint files were missing or recovery crashed.
- The failure-free mismatch strongly indicated changed normal execution semantics.
- The resilient tree’s blast input had been changed away from the original AMR problem, so the clean resilient run could not match the perturbed vanilla baseline.
- The restart flow also used `Initialize(1)` when a VeloC checkpoint existed; that is Athena++ file-restart mode and is unsafe for memory-based VeloC recovery. Recovery now happens after normal initialization.
