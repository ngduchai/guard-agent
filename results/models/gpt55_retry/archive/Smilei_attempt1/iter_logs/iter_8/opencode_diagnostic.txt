Changed `src/Smilei.cpp`:

- Replaced the VeloC memory-protection checkpoint path with VeloC-routed checkpoint files via `VELOC_Route_file`.
- Kept real persistence through libveloc using `VELOC_Checkpoint_begin/end` and `VELOC_Restart_begin/end`.
- Removed all `VELOC_Mem_protect`, `VELOC_Mem_unprotect`, and `VELOC_Recover_mem` usage from `Smilei.cpp`.
- This avoids the likely crash source: VeloC retaining or touching a stack-local `std::vector<char>` buffer after `checkpointSmileiState` returns.
- Added `SMILEI_VELOC_FILE = "smilei_state.bin"` and binary file read/write of the serialized Smilei state.
- Added `<iterator>` for the restart file load path.

Verified existing required plumbing:

- `veloc.cfg` exists at repo root.
- It has absolute, distinct `/tmp` directories and sync mode:

```text
scratch = /tmp/smilei_veloc_scratch
persistent = /tmp/smilei_veloc_persistent
mode = sync
```

- The makefile already includes `/home/ndhai/usr/include`, searches `/home/ndhai/usr/lib`, and links the installed VeloC client library with `-lveloc-client`.

Why this targets the validator failure:

- The previous validator failure was no longer the clean-start `Restart_begin` false positive.
- It was an early segfault in both failure-prone and failure-free resilient runs, with only one partial checkpoint file observed.
- The old path registered a temporary `std::vector<char>` buffer with VeloC for checkpointing. If VeloC touched that buffer after the function returned, the pointer was dangling.
- The new path writes the serialized state to the VeloC-routed checkpoint file during the checkpoint window, so VeloC persists actual state without retaining application memory pointers.
