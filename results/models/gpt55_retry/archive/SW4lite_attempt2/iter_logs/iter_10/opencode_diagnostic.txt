I also found a correctness issue independent of timing: the checkpoint is taken before `exactSol(..., Up, ...)`, which overwrites `Up` later in the same loop iteration. A continuous run enters the next step with that overwritten scratch array, but a restarted run restores the pre-overwrite value; I’m moving checkpointing after that side effect so restart state matches the actual execution boundary.
The checkpoint now happens after the validation-side `exactSol` call. On CPU/no-GPU execution this matters because `Up` is host state and is used again on the next iteration; the restored state must include the same overwrite that a continuous run carried forward.
I’m doing a consistency pass now. The remaining changes are small: fix indentation around the new schedule constants and verify there are no references to the removed 55/65% checkpoints.
One more issue: the final checkpoint at `mNumberOfTimeSteps` happens before the end-of-run validation block, which again overwrites `Up`. If that final checkpoint is selected on future runs it is not a clean terminal state, and it wastes large I/O after output is already determined. I’m removing the final checkpoint from the periodic checkpoint condition.
I removed the terminal checkpoint because the validator kills before the run completes, and the final checkpoint was not needed for recovery while adding significant I/O and another non-identical boundary. The active checkpoints are now early, 45%, and 50%.
Implemented the checkpoint/restart fix in `src/EW.C`.
