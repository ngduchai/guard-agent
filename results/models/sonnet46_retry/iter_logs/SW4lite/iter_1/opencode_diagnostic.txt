## Summary

### Files Created/Modified

**`Makefile`** (modified):

- Added `-I$(VELOC_INC) -DSW4_VELOC` to `CXXFLAGS` (both DEBUG and OPTIMIZE modes)
- Added `-L$(VELOC_LIB) -lveloc-client -Wl,-rpath,$(VELOC_LIB)` to the linker command

**`src/EW.C`** (modified):
- Added `#include "veloc.h"` and `#include <climits>` under `#ifdef SW4_VELOC`
- Added checkpoint/restart logic in `EW::timesteploop`

### Design Decisions

1. **File-based checkpointing via `VELOC_Route_file`**: SW4lite's `cycleSolutionArrays()` rotates underlying data pointers via `Sarray::reference()`. Pre-registering pointers with `VELOC_Mem_protect` would capture stale addresses after each cycle. Instead, I call `U[g].c_ptr()` and `Um[g].c_ptr()` at checkpoint/restart time, getting the current valid pointers.

2. **State captured**: For each grid `g`, both `U[g]` (solution at time n) and `Um[g]` (solution at time n-1) are saved. The scalar state `beginCycle` (next step to run) and `t` (current time) are also saved. This is the minimum state needed to resume the leapfrog time integration.

3. **Checkpoint frequency**: Every 10 time steps and at the final step. This is a reasonable balance between overhead and restart granularity.

4. **`VELOC_Restart_test(..., INT_MAX)`**: Correctly passes a large `needed_version` to retrieve the latest available checkpoint (the API returns the latest version *less than* the given parameter).
