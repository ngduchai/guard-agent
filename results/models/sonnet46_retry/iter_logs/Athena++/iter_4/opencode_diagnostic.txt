The changes are complete. Here's what was fixed and why:

## Root Cause
The previous implementation checkpointed on **every single cycle** (~870 cycles for the blast problem), creating 17,280+ files consuming 13+ GB — exceeding the 10.7 GB disk guard, which killed the process before it could complete.

## Changes Made

### 1. `veloc.cfg` — Added version retention limits
```
max_versions = 1
scratch_versions = 1
```
VeloC will now automatically purge old checkpoint versions, keeping only the latest in both scratch and persistent storage. At peak, this means only ~8 files total (4 ranks × 2 tiers).

### 2. `src/main.cpp` (line 584-593) — Reduced checkpoint frequency
```cpp
// Checkpoint every 20 cycles instead of every cycle
if (pmesh->ncycle % 20 == 0) {
  veloc_scalars[0] = pmesh->time;
  veloc_scalars[1] = pmesh->dt;
  veloc_scalars[2] = static_cast<double>(pmesh->ncycle);
  VELOC_Checkpoint(veloc_name, pmesh->ncycle);
}
```
This reduces the checkpoint cadence from ~870 to ~43 during a full run. With `max_versions=1`, only the single latest checkpoint is retained at any time.

**Disk impact**: From 13+ GB down to ~6 MB (8 files × ~750 KB each), well within the 10.7 GB limit.

**Recovery integrity preserved**: The kill at ~57% runtime (~500 cycles) will land between cycle-480 and cycle-500 checkpoints. `VELOC_Restart_test` finds cycle 480's checkpoint and resumes from there, completing the simulation correctly.
