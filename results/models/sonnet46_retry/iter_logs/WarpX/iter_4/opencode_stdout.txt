## Root Cause Analysis

From the iter_3 validate output, both the failure-prone and failure-free tests failed with **identical** scores (`max_abs_diff=3.332e+10`, `max_rel_diff=1.000e+00`). This indicates both runs produced the same wrong output.

The most likely root cause: **VeloC maintains metadata about checkpoint versions in a location outside the scratch/persistent directories that the validator clears.** When a fresh run starts (or the failure-free run after clearing the checkpoint dirs), VeloC's `VELOC_Restart_test` still returns a valid version number from stale metadata. The subsequent `VELOC_Recover_mem()` fails silently (checkpoint files were deleted), leaving `field_buf` filled with zeros. The `mf_unpack` then zeros out the EM field MultiFabs, causing the simulation to run with zero initial fields. Since `sum(Ex)` is conserved at 0 throughout, the output is all-zero vs. the baseline's ~3.33e10.

## Fix

Added a **sentinel file** mechanism (`/tmp/warpx_veloc_persistent/warpx_ckpt_version.sentinel`):

1. **At checkpoint time**: After each successful `VELOC_Checkpoint_end(1)`, rank 0 writes the current checkpoint version number to the sentinel file inside the persistent directory.

2. **At restart time**: Before calling `VELOC_Restart_test`, rank 0 reads and broadcasts the sentinel file. Only if the sentinel exists (version > 0) AND VeloC confirms the checkpoint exists does the restart proceed.

Since the validator clears `/tmp/warpx_veloc_persistent` before each independent run, the sentinel is also deleted. This ensures:
- **Fresh runs**: sentinel absent → `VELOC_Restart_test` called with `needed_version=1` (guaranteed VELOC_FAILURE) → no restart → correct Gaussian initial condition → correct output
- **Recovery runs**: sentinel present in persistent dir → `VELOC_Restart_test` searches up to `sentinel_version+1` → finds correct checkpoint → restart from correct field state → correct output
- **Failure-free runs**: sentinel cleared by validator → same as fresh runs → correct output
