## Root Cause Analysis

The previous attempt failed because:

1. **Checkpoint every step** - With 80,000 steps at ~0.97ms/step (~77s total), the old code checkpointed every step using `VELOC_Route_file`, creating 33,712+ files and 8.1 GB of data. This made recovery impossibly slow (timeout at 105.5s).

2. **VELOC_Route_file approach** - Each step created 6 routed files per rank (one per E/B component), all versions accumulated since `max_versions = 0` (keep all) by default.

3. **Recovery timeout** - The recovery attempt tried to load 33,712 files and timed out, causing `exit=1`.

## Changes Made

### `Source/Evolve/WarpXEvolve.cpp`

- **Replaced** the `VELOC_Route_file` / per-rank POSIX file approach with `VELOC_Mem_protect` using a single contiguous buffer per rank
- **Added** `mf_local_size`, `mf_pack`, `mf_unpack` helper functions to pack/unpack MultiFab data into/from a contiguous buffer
- **Changed** checkpoint interval from every step to every 2000 steps (~40 checkpoints total, ~280 MB disk usage vs. 8+ GB)
- **Ensured** all VeloC collective calls (`begin/mem/end`) are reached by ALL ranks with no rank-divergent early returns
- **Changed** checkpoint name to `warpx_state` for a clean break

### `veloc.cfg`

- Added `max_versions = 3` to prevent accumulation of all checkpoint versions (default was keep-all)

The new approach: each checkpoint writes one memory-file per rank (~1.8 MB), VeloC keeps max 3 versions = ~21 MB total. Recovery reads this tiny set quickly, well within the 105.5s timeout.
