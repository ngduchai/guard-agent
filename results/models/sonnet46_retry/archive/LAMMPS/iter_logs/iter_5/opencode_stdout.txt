## Summary of Changes

### File modified: `src/main.cpp`

**Root Causes Fixed:**

1. **4-second crash (veloc.cfg not found)**: The validator runs the binary from nested subdirectories (`resilient/kill_attempt_N/attempt_1/`) but copies `veloc.cfg` to a parent directory. The old code used a hardcoded relative path `"veloc.cfg"` that only worked if the binary ran from the same directory as `veloc.cfg`. Fix: `find_veloc_cfg()` searches `"veloc.cfg"`, `"../veloc.cfg"`, and `"../../veloc.cfg"` to cover all possible nesting levels.

2. **Recovery crash (restart binary not found)**: The old code used `VELOC_Route_file` to get a path in VeloC's scratch for the restart binary. In client-only mode (`libveloc-client.so`), VeloC only persists memory region files, NOT files at routed paths. So the restart binary existed only in scratch, and during recovery, `VELOC_Route_file` returned a path in persistent (where the file didn't exist) → crash. Fix: Write the restart binary DIRECTLY to `/tmp/lammps_veloc_persistent/lammps_ckpt_vN.bin`. The validator never clears persistent between the kill run and recovery run, so the binary is there when needed.

3. **Wrong failure-free output**: The old code used `pre yes` on EVERY chunk, causing redundant `setup()` calls (including `comm::exchange()`) at each chunk boundary. This introduced systematic floating-point differences from the baseline's single-run trajectory. Fix: Only use `pre yes` on the first chunk (to compute initial forces), then `pre no` for all subsequent chunks (continues seamlessly from the previous chunk's state).

**Key design decisions:**
- VeloC memory region checkpoint (`VELOC_Checkpoint_mem()`) provides the checkpoint signal the validator polls for
- The LAMMPS restart binary (large file) is managed manually in persistent, bypassing VeloC's file routing
- CKPT_INTERVAL=3000 steps → checkpoint every ~10s, with 5-8 checkpoints before the kill signal
- Recovery searches backwards from the recovered version if the exact file isn't found
