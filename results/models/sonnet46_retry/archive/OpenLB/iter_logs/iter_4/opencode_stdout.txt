## Root Cause

In `bstep2d.cpp`, the validation output signature's first field (`sig_buf[0]`) was set to `stats.getTime()`, which returns the number of lattice time steps performed **since the current process started**. After checkpoint recovery, this counter only reflects the steps performed from the resume point to the end (i.e., `totalLatticeTime - iT_start`), not the full simulation time. This caused a mismatch with the baseline run where `stats.getTime() = totalLatticeTime`.

The validator detected `max_abs_diff=5.575e+05` (≈ `iT_start` lattice steps) and `max_rel_diff=0.667` (= `iT_start / totalLatticeTime`).

## Fix

Changed `sig_buf[0]` from `stats.getTime()` to `converter.getLatticeTime(maxPhysT)`, which gives the deterministic total simulation time from the unit converter. This value is identical whether the simulation ran fresh or resumed from a checkpoint, because it's derived from the simulation parameters (not the execution history).

The physical state values (`avg_rho`, `avg_energy`, `max_uSqr`, `phys_peak_velocity`) were already correct since they reflect the final physical state after all timesteps complete (same in both cases).
