## Summary

### Root Cause of Previous Failure

The previous implementation crashed with `MPI_ABORT on rank 0` (exit code 1). Based on analysis:

1. **GVT hook complexity**: `tw_scheduler_rollback_and_cancel_events_pe` calls `tw_gvt_step2` multiple times internally (in a do-while loop), advancing `me->GVT_sig` and `g_tw_gvt_done` as side effects. This interfered with the outer GVT machinery.

2. **`past_end_time` guard missing**: When GVT reaches `ts_end`, `tw_gvt_hook_step` sets `me->GVT_sig = g_tw_max_sig` and asserts `g_tw_max_sig <= pq_minimum`. If our hook re-injected events into the PQ (making it non-empty), this assert would fire.

3. **Event injection edge cases**: The iter_3 code used manual `tw_eventq_pop` + `memset` instead of the standard `tw_event_grab` API, potentially leaving fields in inconsistent state.

### Fixes Applied (`models/phold/phold.main.c`)

1. **`past_end_time` guard**: Added `if (past_end_time) return;` at the start of `veloc_gvt_hook` to skip checkpointing when the simulation is ending, preventing the assert failure in `tw_gvt_hook_step`.

2. **Standard event allocation**: Used `tw_event_grab(pe)` instead of manual `tw_eventq_pop` + `memset` in `inject_event_into_pq`, ensuring proper initialization of all event header fields.

3. **Cleaner code structure**: Extracted `collect_rng_states()` and `restore_rng_states()` as helper functions for clarity and correctness.

4. **Checkpoint version management**: Added proper error-path version rollback (`g_veloc_version--`) so version numbers stay monotonically increasing even on partial failures.

5. **Proper `VELOC_Checkpoint_end` flow**: Added write-success checking and proper error handling when the event file write fails.
