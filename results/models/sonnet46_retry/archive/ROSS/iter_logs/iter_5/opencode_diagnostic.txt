## Summary of Changes

### Root Cause of Previous Crashes
The previous implementation called `tw_scheduler_rollback_and_cancel_events_pe(pe)` from inside the GVT hook. This function internally calls `tw_gvt_step2` in a loop to drain cancel/event queues. The inner `tw_gvt_step2` calls have destructive side effects on ROSS GVT machinery:
- Advances `me->GVT_sig` 
- Increments `g_tw_gvt_done`
- Triggers fossil collection

These side effects caused GVT state corruption and crashes (MPI_ABORT, exit code 1).

### Fix
1. **`core/tw-sched.c`**: Added `tw_scheduler_rollback_to_gvt_only(pe)` — a new function that:
   - Rolls back all KPs to GVT T (committed state) using `tw_kp_rollback_to_sig`
   - Drains event_q and cancel_q using `tw_sched_event_q` / `tw_sched_cancel_q`
   - Uses `MPI_Allreduce` (via `does_any_pe`) for global coordination
   - Does NOT call `tw_gvt_step2` — no side effects on `me->GVT_sig`, `g_tw_gvt_done`, or `me->gvt_status`

2. **`core/ross-extern.h`**: Added extern declaration for the new function.

3. **`models/phold/phold.main.c`**: Complete rewrite with:
   - Committed-state checkpointing using `tw_scheduler_rollback_to_gvt_only`
   - Checkpointing disabled during recovery runs (`g_veloc_is_recovery` flag)
   - Correct LP RNG state save/restore (including core_rng unconditionally)
   - Event drain/reinject with hash table management
   - Proper VeloC API usage (begin/mem/route_file/end pattern)
   - Recovery path: restores LP RNG states and pending events from checkpoint

4. **`veloc.cfg`**: Already correct — scratch and persistent paths are absolute /tmp paths that differ from each other.
