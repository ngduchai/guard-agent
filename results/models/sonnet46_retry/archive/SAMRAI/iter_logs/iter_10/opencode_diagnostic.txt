Let me do one final check - looking at the `veloc_apply_level_data` lambda. After my edit, it checks `vactual == vsaved.size()` before copying. But for a level-0 patch at the same location, `vactual` (from the current hierarchy after initializeHierarchy) and `vsaved.size()` (from the checkpoint file) should be equal because:
1. The patch box is the same (matched by box key)
2. The ghost width is the same (same input parameters)
3. So the number of cells (including ghost cells) is the same
4. Therefore `getOffset()` returns the same value

For fine-level patches that happen to match (same box position), the same argument holds.

The code is correct and the change is minimal. The only difference from the previous approach is removing the `regridAllFinerLevels` calls during restart.

Here's a summary of the changes made and the reasoning:

## Root Cause

The previous restart approach called `regridAllFinerLevels()` during the restart to rebuild fine AMR levels. This function internally calls `applyGradientDetector()` → `RefineSchedule::fillData()` → `Schedule::communicate()`, which performs MPI communication using schedules built by `initializeHierarchy()`. These schedules are built for the t=0 hierarchy configuration and are inconsistent with the partially-modified hierarchy state during restart, causing a segfault (exit 139) on rank 3.

## Fix

**`source/test/applications/LinAdv/main.cpp`** (lines 386–613):

Removed all `regridAllFinerLevels()` calls from the restart code. Instead, the new approach:

1. Reads all levels' checkpoint data into memory (unchanged)
2. Applies checkpoint data directly to existing patches in the hierarchy by matching box coordinates — **without calling any SAMRAI regridding functions**
3. Restores the integrator's time/step/dt state

The fine-level data from `initializeHierarchy()` (at t=0 configuration) is kept for any patches whose boxes don't match the checkpoint. These will be corrected naturally on the first `advanceHierarchy()` call, which always triggers a regrid (since `regrid_interval=1` for the GRADIENT_DETECTOR strategy), rebuilding fine levels from the correctly-restored level-0 data.

Level-0 patches are always deterministically mapped (same input + same MPI rank count → same boxes), so all level-0 patches match perfectly and get exact checkpoint data. This is the primary state that determines the final simulation output.
