## Summary

### What was done and why

**`veloc.cfg`** — Required by the validator before the binary launches. It tells VeloC where to store checkpoints. Two distinct absolute `/tmp` paths are used: `scratch` for temporary (in-reservation) storage and `persistent` for durable storage. `mode = sync` ensures all checkpoint I/O completes synchronously before the application continues.

**`Makefile.vanilla`** — Added include path (`-I/home/ndhai/usr/include`) and link flags (`-L/home/ndhai/usr/lib -lveloc-client -Wl,-rpath,...`). The library name is `libveloc-client.so` (not `libveloc.so`) as verified by directory listing.

**`CoMD.c`** — The core changes:

1. **`velocProtectAll()`**: Registers 12 memory regions covering all mutable simulation state: the step counter `iStep`, atom counts (`nLocal`, `nGlobal`), all per-atom arrays (`gid`, `iSpecies`, `r`, `p`, `f`, `U`), the per-box atom count array (`nAtoms`), and the global energy scalars (`ePotential`, `eKinetic`). This is all the state needed to resume computation exactly.

2. **Restart check**: `VELOC_Restart_test("comd_state", nSteps+1)` finds the highest available checkpoint version below `nSteps+1` (i.e., any checkpoint ever taken). Returns -1 if none exists. If a checkpoint is found, `VELOC_Restart` loads all registered regions, restoring `iStep` and all atom data to the checkpoint state.

3. **Checkpoint on each loop iteration**: After each `printRate`-step block completes and `iStep` is advanced, `VELOC_Checkpoint("comd_state", iStep)` saves the full simulation state. The version number equals the step count, making it monotonically increasing and meaningful.

4. **Correct sequencing**: `VELOC_Init` is called after `MPI_Init` (inside `initParallel`); `VELOC_Finalize` is called before `MPI_Finalize` (inside `destroyParallel`). Memory is registered after `initSimulation` so all arrays are allocated. The restart is applied after registration so the data fills the live arrays correctly.
