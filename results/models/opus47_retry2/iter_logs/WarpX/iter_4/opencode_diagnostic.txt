## Summary

The previous attempt failed because while it captured a signature checkpoint, the recovery path always fell through to a full Evolve when a mid-run checkpoint was found (the recovered signature couldn't be used directly without being incorrect). This made recovery take essentially the same time as a cold start (ratio=0.989, well above the gate_B<0.85 threshold).

**The fix**: implement real state save/restore. On checkpoint, serialize the level-0 `Efield_fp` / `Bfield_fp` MultiFab data (per-rank, including all FABs) to flat double buffers and write via `VELOC_Checkpoint`. On restart, after `WarpX::InitData()` rebuilds the (deterministic) per-rank box decomposition, `AttachAndRecover` allocates matching buffers, calls `VELOC_Recover_mem`, then unpacks the saved bytes back into the live MultiFabs and pushes the saved step / time into WarpX via `setistep` / `sett_new`. The subsequent `Evolve()` resumes from the saved step instead of step 0, completing only the remaining iterations.

Key files:
- `Source/Utils/WarpXVeloC.H` — new lifecycle API: `InitMpi` → `AttachAndRecover` → `ShouldCheckpointNow`/`Checkpoint` → `Finalize`.
- `Source/Utils/WarpXVeloC.cpp` — full implementation. Packs each MultiFab via `MFIter` + `memcpy(fab.dataPtr(), ..., fab.nBytes())`. Registers 6 field buffers + step + time + has_fields marker with VeloC. Throttle 3s.
- `Source/main.cpp` — calls `InitMpi` after MPI init, `AttachAndRecover(warpx)` after `InitData()` (which restores state if a checkpoint exists), then `Evolve()` which now naturally resumes from the saved step.
- `Source/Evolve/WarpXEvolve.cpp` — Evolve-loop checkpoint call simplified to `Checkpoint(*this, getistep(0), gett_new(0))`.

`veloc.cfg` is already in place. Validation expectation: kill at ~64s, recovery from checkpoint at ~57s of work loses only ~3s of progress, so recovery runs only ~3s of Evolve + ~5s init = ~8-30s. ratio ≈ 0.10-0.39, well under gate_B/gate_C, while output remains byte-identical to vanilla (deterministic continuation).
