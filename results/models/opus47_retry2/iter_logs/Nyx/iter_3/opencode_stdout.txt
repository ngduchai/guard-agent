The previous iteration failed because on restart, the recovered binary aborted with `amrex::Error::get_comoving_a: invalid time` (exit code 6) within ~3s. The validator log clearly showed checkpoint files (20 files, 6.9MB) were being written and a restart attempt was made, but the binary aborted.

**Root cause**: `Nyx` keeps cosmology bookkeeping as file-scope `static Real` members: `old_a`, `new_a`, `old_a_time`, `new_a_time` (plus `initial_z`, `final_a`, `final_z`, `initial_time`, `final_time`). These are advanced by `integrate_comoving_a()` at each coarse step. After `amrptr->init(0, stop_time)` on restart, they get re-initialised to `0/0/1/1` for `old_a_time/new_a_time/old_a/new_a`. The checkpoint code then loaded the saved cumtime (~1.4 sec) but did NOT restore these statics, so the next call to `get_comoving_a(cur_time=1.4)` failed the bracketing check at `comoving.cpp:519` and called `amrex::Error("get_comoving_a: invalid time")`.

**Fix** (in `Source/Driver/nyx_main.cpp`):
1. Bumped `NYX_CKPT_VERSION` from 2 → 3.
2. In `writeRankCheckpoint`, after writing `cumTime`, also write the nine cosmology statics.
3. In `readRankCheckpoint`, after reading `saved_cumtime`, read those nine doubles; then after `setCumTime`, assign them back into `Nyx::old_a`, `Nyx::new_a`, `Nyx::old_a_time`, `Nyx::new_a_time`, `Nyx::initial_z`, `Nyx::final_a`, `Nyx::final_z`, `Nyx::initial_time`, `Nyx::final_time`.

This preserves the actual state-capture semantics (real persisted scalars that drive the next timestep's cosmology lookup), so recovery resumes correctly rather than aborting on the first source-term evaluation.
