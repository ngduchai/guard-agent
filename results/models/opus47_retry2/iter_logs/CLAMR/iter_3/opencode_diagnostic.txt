I've made the critical fix. Summary of changes:

**Problem identified:** In iter_2, recovery took 91% of failure-free runtime instead of the expected ~34%. Output matched (because CLAMR is deterministic), but the restore code path was never actually taken — the recovery binary ran from cycle 0.

**Root cause:** `VELOC_Restart_test(name, 0)` was being called to look for a checkpoint. According to the veloc.h documentation, this function returns "latest version < needed_version". With `needed_version=0`, no checkpoint can have version < 0, so it always returned `VELOC_FAILURE (-1)`. This caused the restore block to be skipped entirely.

**Fix:** Changed to `VELOC_Restart_test(CLAMR_CKPT_NAME, INT_MAX)` to retrieve the latest checkpoint of any version. Added `#include <climits>` for `INT_MAX`. Also added diagnostic `printf` statements so future debugging can see what the function returned.

The fix is minimal and surgical. With the restore actually executing:
- Saved cycle (~4400 out of 6700) will be restored
