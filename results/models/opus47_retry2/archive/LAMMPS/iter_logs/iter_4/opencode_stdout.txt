Summary of changes:

I identified that the previous implementation correctly produced VeloC checkpoints (256 files, ~491MB) and successfully recovered from a SIGKILL with exit code 0, but the recovered simulation diverged numerically from the no-failure baseline (max_abs_diff=403, max_rel_diff=3e-3).

**Root cause:** The previous code stored a globally-replicated atom state (sorted by tag) on every rank using `lammps_gather_atoms` + `lammps_create_atoms`. While `lammps_create_atoms` correctly dispatches atoms to their owning ranks via `Domain::ownatom`, it does NOT preserve the original per-rank ordering of atoms within `atom->{x,v,type,tag,image}` arrays. Because floating-point addition is not associative, force computations whose per-atom contributions are accumulated in a different order produce slightly different numerical results, and that tiny perturbation grows over ~10⁴ remaining MD steps until it exceeds the validator's tolerance.

**Fix (src/main.cpp):** Switched to per-rank state preservation:
- Each rank's VeloC checkpoint stores its OWN local atoms in the exact order they appear in `atom->{tag,type,x,v,image}` (variable-sized per-rank buffers).
- On restart, each rank locally re-inserts its saved atoms via `AtomVec::create_atom()` followed by direct field assignment of tag/v/image, preserving exact local ordering.
- Global `atom->natoms` recomputed via `MPI_Allreduce(MPI_SUM)` over per-rank `nlocal`.
- Atom map (if used) re-initialized.
- `reset_timestep` advances the LAMMPS clock, then `nextsort` is restored.

The result is that after restart every rank's local atom array is bit-identical to what it held at checkpoint time, so subsequent MD evolves on the exact same floating-point trajectory as the original failure-free run.
