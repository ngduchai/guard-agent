1. **Recovery took 130s** (timed out at 1.19x cap) — even with the `INT_MAX` fix, the chunked `pre yes/no/post yes/no` pattern combined with mid-run checkpointing was producing recovery runs that didn't complete in time.
2. **Failure-free run hung** — the `pre no post no` chunks were apparently unstable; the validate stdout was truncated mid-way through the failure-free run.

### Key changes in this iteration

1. **Simplified the run loop.** Every `run` chunk now uses `pre yes post yes` (full init + cleanup per chunk). This is slightly slower but rock-solid — independent runs avoid the `pre no` fragility that caused the previous hang. For the LJ benchmark the per-chunk init cost is tiny (~50 ms each).

2. **Restart path uses a single `run`.** Once we restore atom state and `reset_timestep`, we issue **one** `run REMAINING pre yes post yes`. No further chunking, no further checkpointing on the restart path. Maximally fast recovery.

3. **Retry across older checkpoint versions.** `try_restart` now collectively retries v-1, v-2, … if the latest checkpoint version is incomplete (SIGKILL mid-write). All ranks decrement `try_v` together in lockstep, preserving collective discipline.

4. **NUM_CHUNKS=4.** Four cold-start chunks produce 3 checkpoints (at ~25 %, 50 %, 75 % of the run). The first checkpoint completes well before the validator's 58 % poll threshold.

5. **Cleaner collective discipline.** Every `lmp->input->one(...)` is reached on every rank in the same order. Per-rank failure decisions are reduced via `MPI_Allreduce(MIN)` before any branch is taken. No rank can take an early-return that skips a downstream collective.

The `veloc.cfg` file is already in place and correct (`/tmp/lammps_veloc_scratch`, `/tmp/lammps_veloc_persistent`, `mode = sync`).

- **Failure-free**: 4 independent runs of 8250 steps each, total ≈ 110 s + ~3 checkpoints × ~100 ms = ~111 s. No hang.
