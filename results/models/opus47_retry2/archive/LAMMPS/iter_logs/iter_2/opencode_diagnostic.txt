Summary of changes from previous attempt:

1. **Replaced gather/scatter approach with per-rank serialization**: The previous approach used `lammps_gather_atoms` (collective, full N atoms on every rank) followed by `lammps_scatter_atoms` on restart. This had several issues:
   - Atoms were placed on ranks based on initial-lattice positions, not checkpoint positions
   - `comm->exchange()` during setup couldn't handle multi-subdomain migration
   - Atom local ordering differed from failure-free run → different force accumulation order → numeric divergence

2. **New per-rank approach**: Each rank:
   - Serializes its OWN local atoms (tag, image, x, v) in current local order
   - On restart, clears its atoms and recreates them from checkpoint in the EXACT order they were when checkpointed
   - This preserves per-rank atom ownership and local indexing, ensuring bit-identical force computation order

3. **Added preservation of `atom->nextsort`**: This is the LAMMPS-internal counter for the spatial sort schedule. Without restoring this, the sort schedule diverges from failure-free, causing atom reordering at wrong timesteps and subsequent numerical drift.

4. **Reduced checkpoint size**: From ~470MB to ~110MB by avoiding the gather-replicate-on-all-ranks pattern.

5. **Updated CMakeLists.txt** to also search `$HOME/usr` for veloc headers/libs (in case the system's CMake searches don't find it).

The key insight is that determinism in MPI floating-point reductions requires preserving:
- Which atoms each rank owns (decomposition state)
- The local ordering of atoms on each rank (force accumulation order)
- Internal LAMMPS scheduling state like `nextsort` (event timing)

The veloc.cfg file remains valid (absolute /tmp paths, scratch ≠ persistent, sync mode).
