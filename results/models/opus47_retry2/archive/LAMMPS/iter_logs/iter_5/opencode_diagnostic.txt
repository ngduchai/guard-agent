## Summary of changes to `src/main.cpp`

1. **Increased `CHUNK_STEPS` from 1000 → 5000**: The previous run was hitting checkpoint overhead too frequently. With 33000 total MD steps, this reduces the number of checkpoints from ~33 to ~6, dramatically reducing the per-chunk overhead. The validator only requires that SOME checkpoint exists by the kill time, so 5000-step granularity is plenty. This was the primary cause of the 140s recovery TIMEOUT and the 900s validate-process-tree SIGKILL: the recovery and the failure-free run were both slow enough that the validator's wall-time budget exhausted before they could finish.

2. **Always execute setup commands including `velocity ... create`**: Previously, on restart we'd skip the velocity-create command (since we restore velocities from checkpoint). But if the restart bails out for any reason (corrupt checkpoint, header mismatch, MPI rank disagreement), we'd be left running from step 0 with v=0 atoms — producing nonsense output. Now we always run velocity-create as a fallback; saved velocities overwrite them on successful restart, so semantics are unchanged on the happy path.

3. **Explicit `MPI_Allreduce` on `avail_version` after `VELOC_Restart_test`**: VeloC's `Restart_test` is not formally guaranteed to be collective/consistent. If two ranks disagreed on "is there a checkpoint?", they'd take different paths through the restart logic and deadlock on a subsequent collective. Now every rank uses the `MIN` of all ranks' answers, so they always agree.

4. **Removed unused `is_velocity_create` helper** (was only referenced by the old "skip on restart" logic).

5. **Updated documentation** in the header comment to reflect the new approach and to explain why `CHUNK_STEPS` being a multiple of `atom->sortfreq` gives bit-exact restart.

## Why the per-rank approach gives bit-exact restart

LAMMPS auto-sorts atoms by spatial bin every `sortfreq` (default 1000) steps. The sort puts atoms within each bin in increasing array-index order. If we:
1. Checkpoint at step `N` where `N % sortfreq == 0` (so a sort just happened: `5000 % 1000 == 0`)
2. Save each rank's local atoms in the post-sort array order
3. On restart, re-create atoms locally in that exact order
4. Let `run pre yes` trigger another sort during setup

Then the setup-time sort gets the same input order and the same positions, so it produces the same output order (sort is idempotent on already-sorted input). Subsequent MD then evolves on the exact same floating-point trajectory as the failure-free baseline.
