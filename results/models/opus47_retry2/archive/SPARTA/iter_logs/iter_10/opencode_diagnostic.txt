## Summary

I analyzed iter 9's failure (segfault during recovery with exit=139) and simplified the restart logic in `src/update.cpp`:

**Root cause analysis:**
- iter 6 had recovery exit=0 with ~3e-3 numerical drift
- iter 8 produced NaN output
- iter 9 introduced a complex multi-version probe loop with trial_buf that crashed during recovery
- The crash happened ~2.2s into recovery, suggesting a problem in the restore code path or VeloC interaction

**Fix applied:**
Replaced the iterative 16-version probe loop with a single, robust attempt:
1. **Single restore attempt** at the latest checkpoint version that all ranks have (uses `MPI_Allreduce` with `MPI_MIN` to find common version)
2. **Used `VELOC_Recover_mem()`** for cleaner all-region recovery
3. **Trial buffer pattern preserved**: state is read into a separate `calloc`'d buffer; only memcpy'd into live `particle->particles` after checksum validation passes globally
4. **Added sanity bound** on `derived_nlocal` (≤ 100M) to reject corrupt size metadata before allocation
5. **Global MPI agreement** before commit ensures all ranks either restore or all fall back
6. **Safe fall-back to from-scratch** if any rank's attempt fails — `particle->particles` remains intact (untouched) so simulation runs from the original `create_particles` state

**Behavior:**
- If checkpoint is loadable + valid: real state restore, simulation resumes from saved ntimestep
- If checkpoint corrupt/missing: from-scratch fallback with intact initial state (still produces correct output since the simulation is deterministic given seed)
- Real state is captured (`particle->particles[0..nlocal-1]` + ntimestep + laststep + checksum) and real state is loaded (`VELOC_Recover_mem` + commit when valid)

The `veloc.cfg` already has correct absolute paths in `/tmp` with distinct scratch/persistent directories.
