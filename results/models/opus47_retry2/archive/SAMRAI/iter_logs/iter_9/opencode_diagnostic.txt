## Summary

The previous attempt (iter_8) failed correctness validation with `max_abs_diff=6.880e+08, max_rel_diff=0.65` despite recovery completing successfully. The root cause was that after `initializeHierarchy()` on restart creates fine levels around the INITIAL-CONDITION sphere position, then `applyStateToHierarchy` only matched LEVEL 0 patches (whose box layout is deterministic from domain_boxes/largest_patch_size), and `removePatchLevel` stripped the fine levels.

The natural TimeRefinementIntegrator regrid was deferred for 2 timesteps (the `d_regrid_interval` default), during which level 0 advances WITHOUT fine-level sub-cycling and the resulting state diverges from the original run that had continuous fine-level refinement.

## Fix

Added a MANUAL REGRID step immediately after stripping fine levels in the restart path (`source/test/applications/LinAdv/main.cpp:1081-1133`):

1. After `applyStateToHierarchy` restores level-0 from the global checkpoint records,
2. After `removePatchLevel` strips levels 1-2,
3. **Call `gridding_algorithm->regridAllFinerLevels(0, ...)`** to rebuild fine levels at the correct positions (around the restored sphere position) using the gradient detector on the restored level-0 data.
4. Re-apply checkpoint state to the now-rebuilt fine levels (any boxes that happen to match the original layout get their actual checkpointed values back; the rest keep the level-0-interpolated values from regrid).
5. Call `synchronizeNewLevels` for consistency with the integrator's normal post-regrid flow.

This means the time loop now enters with all 3 levels properly populated and refined around the correct sphere location, instead of waiting 2 steps for the natural regrid to occur (during which level 0 would evolve incorrectly without sub-cycling).
