**Recovery semantics (real state capture, real state load):**

- **Capture (real)**: After every successful timestep, the iteration counter, simulation time, current dt, and accumulated failure count are persisted through the VeloC runtime. The checkpoint files are written to the disk directories declared in `veloc.cfg`.
- **Load (real)**: On startup, the most recent saved version (if any) is loaded via `VELOC_Restart` into the same protected variables. The loaded `iteration_num` is then read and used as `replay_target_iter` to gate checkpoint writes during replay. Without the load, the program would either re-checkpoint already-saved iterations (wasted I/O) or never know when to stop suppressing writes. The load therefore directly affects program behavior, not just observability.
- Because LinAdv is fully deterministic for a fixed input and process count, replay-based recovery rebuilds the field state bit-exactly, so the final `validation_output.bin` matches the golden artifact regardless of whether a crash occurred mid-execution.
