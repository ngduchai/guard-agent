  - This creates a useful mid-run checkpoint before the observed kill point and avoids recovering from a very early checkpoint that still takes almost a full run.

    - grow particle storage,
    - register particle buffer,
    - recover particle data.
  - Added `veloc_sparta_checkpoint_available()` so other setup code can detect an existing VeloC checkpoint.

  - On restart, skips expensive `create_particles` generation when VeloC reports a valid checkpoint.
  - Recovery still happens at `run`, after static setup commands recreate grid/species/mixture/stats, but before timestepping resumes.

- `veloc.cfg`
  - Kept required static config in source root with absolute `/tmp` paths.
  - Increased bounded retention to:
    - `scratch_versions = 2`
    - `max_versions = 2`
  - This preserves the useful mid-run checkpoint while still keeping disk usage bounded.

Why this addresses the failure:
- The previous validator output showed output correctness passed, but `recovery_actually_resumed=False` because recovery took about the same time as a full run.
- The app was replaying expensive input setup, especially particle creation, before reaching restart logic.
- The old `5000` timestep checkpoint cadence also meant the recovered run replayed nearly the whole simulation.
- Now restart skips particle recreation and resumes from a later real VeloC checkpoint containing timestep, global/local particle counts, and the local particle array.

I could not run a local build or validation because no shell execution tool is available in this environment, but I performed targeted static checks of declarations, includes, config, and call sites.
