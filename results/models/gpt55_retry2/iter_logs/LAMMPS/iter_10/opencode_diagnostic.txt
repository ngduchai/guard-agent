What changed:
- Fixed the constructor bug where `VELOC_Restart_test()` computed `min_latest` but never assigned it to `restart_step`. This could prevent VeloC metadata recovery from being attempted.
- Added a direct persisted state fallback under VeloC’s configured persistent directory: `/tmp/lammps_veloc_persistent/direct`.
- The direct fallback writes the same real checkpoint payload as the VeloC path: timestep, global atom count, local atom count, and per-rank atom positions, velocities, tags, types, masks, and images.
- Added atomic-ish write/publish behavior:
  - State files are written via temp files then renamed.
  - Per-rank `done` markers are written only after state files complete.
  - `latest_rank*.bin` markers are published only after all ranks successfully complete the checkpoint/VeloC path.
- Recovery now first tries the completed direct persisted state for the selected version, then falls back to `VELOC_Restart_begin()` plus routed file recovery and registered-memory recovery.
- Reduced the default checkpoint interval from `500` to `250` timesteps, improving the chance that a recent complete checkpoint exists before the validator’s kill point.
- Verified `veloc.cfg` exists at repo root and uses required absolute `/tmp` paths:
  - `scratch = /tmp/lammps_veloc_scratch`
  - `persistent = /tmp/lammps_veloc_persistent`
  - `mode = sync`

Reason for the fix:
- The previous validation failed because output was correct but recovery took nearly a full run: `recovery/Z_P=1.013`, so the validator concluded it did not actually resume.
- The missing `restart_step = min_latest` assignment meant VeloC metadata restart could be skipped.
- The added completed-version marker fallback protects against hard-kill cases where VeloC metadata or routed files for the newest checkpoint are incomplete/unavailable while still retaining real VeloC checkpoint calls and real state load.
