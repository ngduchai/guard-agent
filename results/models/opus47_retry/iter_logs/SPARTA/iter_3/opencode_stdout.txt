The fix is in place. Summary:

**Problem**: Recovery resulted in `validation_output.bin` with NaN values. Test 1 (failure-prone correctness) failed.

**Root cause**: The single-pass `VELOC_Recover_mem()` restoration in `Update::run()` registered the particles region (ID=2) with a slot count derived from `VELOC_Recover_size(2)` called *before* any data was unpacked. In some VeloC implementations this returns -1 (or 0) when the id has not yet been observed in the current restart phase, leaving the particles buffer protected with only 1 slot. `VELOC_Recover_mem()` would then either overrun the buffer (corrupting memory and producing NaN in the final reduction) or silently skip the particle region (so the freshly-created seed-perturbed particles persisted at step 0 instead of being advanced to the saved step), either way producing a wrong output that fails Test 1.

**Fix**: Switch to a two-phase recovery in `src/update.cpp`:
1. Protect IDs 0 (`ntimestep`) and 1 (`nlocal`); call `VELOC_Recover_selective(VELOC_RECOVER_SOME, {0,1}, 2)`.
2. Use the now-known `saved_nlocal` (cross-checked against `VELOC_Recover_size(2)`) to grow `particle->particles` to the correct capacity, protect ID 2 with the exact slot count, then call `VELOC_Recover_selective(VELOC_RECOVER_SOME, {2}, 1)`.

This guarantees the particle array buffer is always exactly the right size before VeloC writes into it, so no overrun and no silent skip. The rest of the checkpoint/restore logic (cumulative_target accounting, periodic checkpointing every 2000 steps, file purging) is unchanged.
