**Changes to `/home/ndhai/diaspora/guard-agent/build/tests_baseline_opus47_retry/HyPar/src/Simulation/Solve.cpp`:**

1. **Wall-clock-paced checkpoints** (every ~8 seconds with a first one at ~2s) instead of iteration-based (every iter ≤5, then every 50 iters). The original cadence produced ~94K checkpoints per 80s wall time at the test problem's ~17 µs/iter rate, generating hundreds of thousands of files in `/tmp/hypar_veloc_*`. That's what caused VELOC_Restart_test to scan forever during recovery and the watchdog to fire.

2. **Collective decision** for whether to checkpoint: rank 0 polls the wall clock and `MPI_Bcast`s the decision. Polling is gated by a `CKPT_POLL_STRIDE` of 256 iterations to keep the per-iteration overhead negligible.

3. **Collective decision** for restart version: all ranks compute `VELOC_Restart_test` locally, then `MPI_Allreduce(MIN)` to agree on the version they will all restart from.

4. **Collective decision** for restart success: every rank reports its `VELOC_Restart` return code via `MPI_Allreduce(LAND)`. We only declare restart successful if every rank succeeded, avoiding the deadlock pattern where some ranks have restored state and others have not.

5. **Monotonic version numbers** that survive restart: `_veloc_ckpt_seq` is seeded from the resumed `ckpt_start_iter`, and we always pick a version strictly greater than the previous one written by this process.

The `veloc.cfg` was already correct (absolute distinct `/tmp` paths, `mode = sync`), so no change there.

The previous iteration's deadlock/timeout came from VeloC backend taking too long to enumerate/manage 282 460 checkpoint files inside the 87.8s recovery budget; reducing the checkpoint cadence from ~once-per-iteration to once-per-8s collapses that file count to ~10 per rank and the recovery should complete in well under the budget.
