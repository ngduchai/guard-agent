3. **`./clamr_mpionly.cpp`**: Added VeloC-based checkpoint/restart:
   - Includes `<veloc.h>` when `HAVE_VELOC` is defined.
   - In `main()` after `L7_Init`, calls `VELOC_Init(MPI_COMM_WORLD, "veloc.cfg")` and probes for an existing checkpoint via `VELOC_Restart_test`.
   - After the initial mesh/state are built and `calc_neighbors_local` is called, if a checkpoint exists, calls `read_checkpoint()` to overlay the saved state (i, j, level, H, U, V arrays plus ncycle/simTime/deltaT/next_graphics_cycle/H_sum_initial/upper_mass_diff_percentage), then rebuilds derived data (celltype, spatial coords, neighbors).
   - At the end of each `do_calc()` burst (after ncycle has advanced), calls `write_checkpoint(ncycle)` to persist a fresh checkpoint with version=current cycle.
   - `write_checkpoint` and `read_checkpoint` use `VELOC_Route_file` to obtain a per-rank file path, then `fwrite`/`fread` a binary payload with magic + format version + scalars + the variable-length AMR arrays. This is required because the AMR mesh size changes every cycle so fixed-size memory protect doesn't fit.
   - On clean exit calls `VELOC_Finalize(0)`.

The key design choice is the file-based payload (via `VELOC_Route_file`) rather than `VELOC_Mem_protect`: CLAMR's AMR makes `ncells` (and therefore `i/j/level/H/U/V` extents) change every iteration. Re-protecting after every resize would be fragile; routing one file per rank and serialising the payload ourselves is simpler and avoids stale pointer/size protections. The restore path rebuilds neighbours/celltype/spatial coords from the restored primary AMR data (i, j, level), which is correct because those are derived quantities.
