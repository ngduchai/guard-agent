Summary of changes:

The previous implementation routed per-rank data files through `VELOC_Route_file` and relied on VeloC to manage the persistence/recovery of those files. The validation showed that recovery walltime equaled baseline walltime (ratio=1.002), proving the restart path silently fell back to a fresh run from t=0 instead of resuming from the checkpoint.

**Fix in `src/main.cpp`** (full rewrite of the checkpoint/restart machinery):

1. **Parse `persistent` from `veloc.cfg` at startup** and use it as the absolute home for the LAMMPS data file (no longer routed per-rank).

2. **Single shared data file written to an absolute path** in the persistent dir under a versioned name (`<persistent>/lmp_ckpt_state_v<N>.data`). Rank 0 writes via `write_data`, then atomically renames `.tmp → final` so a kill mid-write cannot leave a half-written file at the canonical path.

3. **VeloC owns only the tiny 3-long header** `{ntimestep, total_steps, data_version}`. Because the header is committed atomically by `VELOC_Checkpoint_end`, finding a valid header on restart implies the corresponding data file exists.

4. **On restart**: `VELOC_Restart_test` → MPI_MIN across ranks → `Recover_selective` reads the header → rank 0 verifies the data file exists at the absolute path, stages it as `./lmp_ckpt_state.data`, and we run `read_data` + `reset_timestep <step>` to put LAMMPS back at the saved step. Then chunked run continues with `remaining = total_steps − start_step`.

5. **Per-rank failure modes are MPI_Allreduce'd** before any divergent control flow.

This removes the previous failure mode where VeloC's per-rank route/registration apparently dropped the data file between attempts, causing restart to silently fall through to fresh execution.
