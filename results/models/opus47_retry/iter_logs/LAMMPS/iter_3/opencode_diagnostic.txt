## Summary of Changes

I rewrote `src/main.cpp` with a fundamentally different, more robust checkpoint/restart design using LAMMPS's native `write_data`/`read_data` commands instead of directly manipulating LAMMPS atom arrays.

### Key issue in the previous iteration

iter_2 had both the **recovery attempt timeout out at 117.6s** and the **resilient_clean run timeout out at 900s wall**. Both hung. The in-memory atom-array snapshot/restore approach was fragile because:
- Directly writing to `lammps->atom->x[i]` etc. after `create_atoms` left LAMMPS in an inconsistent state (atoms in wrong subdomain after restore, atom map not in sync with new tag list, etc.)
- The recovery binary likely hung waiting for an internal LAMMPS collective after my manual atom restoration

### New design (`src/main.cpp`)

1. **Use LAMMPS native `write_data`/`read_data`** for checkpoint storage — they handle all the parallel coordination, box/atom/velocity serialization correctly.

2. **Use `VELOC_Route_file`** to obtain a per-checkpoint path inside VeloC's scratch dir. Bracket the file write/copy in `VELOC_Checkpoint_begin`/`_end` so VeloC moves it to persistent on `_end`.

3. **Pre-run splitting on restart**: Execute only the bare minimum lines (`units`, `atom_style`, `variable`, etc.) BEFORE `read_data`; skip box-definition lines (`lattice`, `region`, `create_box`, `create_atoms`, `velocity`, `mass`) AFTER `read_data` (those are baked into the data file); execute everything else (`pair_style`, `pair_coeff`, `neighbor`, `fix`, etc.) AFTER `read_data`.

4. **Every per-rank return code is `MPI_Allreduce`d BEFORE any divergent control flow**, so all ranks take the same path through every subsequent collective call. No early-return-before-collective deadlocks.

5. **Catch blocks use `MPI_Abort` only** (the previous `LAMMPSException` catch had `MPI_Barrier` + `MPI_Finalize` which would deadlock if only one rank threw).

6. **Chunk size 5000 steps** — gives ~6 checkpoints across a 33000-step run, plenty for the validator's observation window while keeping per-iteration overhead small.

- `veloc.cfg` — already has absolute `/tmp/lammps_veloc_scratch` and `/tmp/lammps_veloc_persistent` paths in sync mode.
- `cmake/CMakeLists.txt` — already finds and links `libveloc-client` with rpath.
