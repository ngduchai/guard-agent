## Summary

The previous iteration (iter_9) used `write_data` + `read_data` for checkpoint/restart. The recovery completed cleanly (exit=0, ratio=1.05x — well within the 1.2x cap), but the post-restart trajectory diverged from the perturbed-vanilla baseline by ~302 in coordinate sums (max_rel_diff=2.823e-03), failing the bit-exact numeric-tolerance gate.

**Root cause**: `read_data` redistributes atoms across MPI ranks by spatial subdomain ownership, producing a NEW within-rank array order on restart. Even though the side-state binary restored bit-exact x/v/image by tag, the local-array order changed → neighbor list traversal order changed → pair-force summation order changed → tiny ULP-level rounding differences → chaos-amplified divergence over the remaining ~13 000 integration steps. The previous `failure-free` test passed bit-exact (no restart, just chunked-run with `write_data noinit` between chunks), confirming the divergence is purely a restart-path artifact.

**Fix**: Bypass `read_data` entirely on restart. New flow:
1. Per-checkpoint, each rank writes a per-rank binary side-state file (`lmp_side_v<V>_rank<R>.bin`) containing tag, type, image, x, v in the rank's CURRENT local-array order — fully bit-exact.
2. A tiny VeloC-managed header carries `(ntimestep, total_steps, data_version, world_size)`.
3. On restart, execute the pre_run script lines, **skipping only the atom-creating lines** (`create_atoms`, `read_data`, `velocity`). Keep `lattice`/`region`/`create_box`/`mass`/`pair_style`/`pair_coeff`/`neighbor`/`neigh_modify`/`fix`. After pre_run, the box exists and `atom->nlocal == 0`.
4. Each rank loads ITS OWN side-state file and manually populates its atoms via `avec->create_atom(type, coord)` + direct assignment of tag/image/v in the EXACT local-array order from the checkpoint. `atom->natoms` is updated via `MPI_Allreduce`; the atom map is rebuilt if any.
5. `reset_timestep`, then continue the chunked `run` loop from the saved step.

**Critical alignment**: `CKPT_CHUNK_STEPS = 1000`, which equals `atom->sortfreq` (default 1000) AND is a multiple of `neigh_modify every 20`. So checkpoints land on iterations where the no-restart run has just performed `PBC + comm->exchange + atom->sort + comm->borders + neighbor->build`. On restart, `Verlet::setup()` re-runs exactly that same sequence — `atom->sort()` is forced unconditionally regardless of `nextsort`, putting atoms back into the same deterministic spatial-bin order that the no-restart run had at this step. From there, integration continues bit-exactly.

`veloc.cfg` is already in place at the source-tree root with absolute `/tmp` paths (`scratch=/tmp/lammps_veloc_scratch`, `persistent=/tmp/lammps_veloc_persistent`, `mode=sync`) — unchanged from iter_9.
