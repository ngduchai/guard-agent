## Summary

**Root cause identified from iter_7 logs**: Test 1 (failure-prone) failed numeric-tolerance with `max_rel_diff=7.447e-03`. The previous iteration's read_data + binary-overwrite approach was insufficient because **`read_data` rebuilds the local atom array in *file order*** (atoms appear in tag-sorted order, distributed by spatial ownership), not in the *migration-history order* the in-memory continuous (Z_P) run had at the same step. Different local-array order → different neighbor-list iteration order → ULP-level differences in pair-force accumulation per step → Lyapunov-amplified macroscopic divergence over 10000+ post-restart steps.

**Fix in src/main.cpp**: Replace the read_data path entirely.

1. **Checkpoint format**: write one binary file per rank (`lmp_ckpt_state_v<V>_rank<R>.bin`) containing the rank's local atom array (`tag`, `image`, `type`, `x[3]`, `v[3]`) in its **current local-array order**. Plus a small meta file with box dims and masses. All writes go via .tmp + atomic rename(2).
2. **Restart path**:
   - Re-execute every pre-`run` script line EXCEPT the three "atom-creating" lines (`create_atoms`, `read_data`, `velocity`). That sets up box, force field, neighbor config, NVE fix with zero atoms.
   - Populate each rank's local array directly via `avec->create_atom(type, x)` in the saved order, then fix up `tag/image/v` per record. This is the only way to preserve the within-rank ordering that the continuous run produced through its migration history.
   - Allreduce `atom->natoms`, rebuild the global tag→local map, `tag_check`, `reset_timestep`.
   - **Temporarily set `atom->sortfreq = 0`** then issue `run 0 pre yes post no` so LAMMPS performs `init() + setup(1)` (exchange, neighbor build, force compute) **without** calling `atom->sort()`. Restore `sortfreq` and align `atom->nextsort` with vanilla's schedule (`(T/sortfreq)*sortfreq + sortfreq`) so subsequent in-run sorts happen at the same step numbers vanilla sorts at.
   - In the chunked run loop, use `pre no` on the first chunk after restart (setup already done).

This produces a bit-exact resume: the first force calc after restart sees the same `(i, j)` pair iteration order vanilla saw, so the trajectory stays identical to the un-killed baseline through step 33000.
