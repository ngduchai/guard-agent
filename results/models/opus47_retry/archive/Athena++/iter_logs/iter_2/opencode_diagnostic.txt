Actually maybe — VeloC writes to scratch first, then copies to persistent. If killed mid-copy, persistent has partial files. Restart might succeed but with corrupted data. That would explain the output mismatch.

The fix for this: ensure VELOC_Checkpoint completes before kill. The validator does `waiting 5.0s to let the in-flight checkpoint write complete before SIGKILL`. So they explicitly wait 5s. With mode=sync, our VELOC_Checkpoint blocks until done. So 5s after first checkpoint file appears should be enough... unless we start ANOTHER checkpoint right after.

With CKPT_INTERVAL=10 cycles (~64ms), checkpoints happen super frequently. So after each ckpt completes, immediately starts next. When validator waits 5s, many more ckpts start and partially complete. Kill might catch one mid-write.

With CKPT_INTERVAL=500 cycles (~3.2s), checkpoints happen every 3.2s. The 5s wait is more than enough to let one complete. Kill happens when no ckpt is in flight. Restart reads complete data. 

This is another reason CKPT_INTERVAL=500 is good!
