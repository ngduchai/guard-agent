Summary of changes made to fix the recovery-doesn't-actually-resume bug from iter_2:

**Root cause of iter_2's failure**: 
- `recovery_actually_resumed=False` because `ratio=recovery/Z_P=1.015` — recovery took as long as a full fresh run, meaning the resumed process re-did all the work instead of fast-forwarding from the checkpoint.
- Symptom: `failure_prone` checkpoint metrics showed only 3360 bytes / 8 files (= 420 bytes/file ≈ meta header only, no state buffers), while `failure_free` had 5MB/8 files. State buffer registration likely had issues in the recovery process.

**Fix in `Source/Driver/nyx_main.cpp`**:

1. **Single-region blob layout** — Replaced the two-phase restart (meta header + per-(level,state) buffers) with a single per-rank byte blob registered under VeloC id 0. The blob is `[NyxVelocHeader][packed state doubles]`. This makes the checkpoint atomic, avoids two-phase recover_selective semantics, and uses `VELOC_Recover_size(0)` to learn the exact size at restart time.

2. **Magic number validation** — Added `NYX_VELOC_MAGIC = 0x4E594F58` ('NYOX') to the header so a partially-recovered or zeroed blob is detected as invalid and we fall back to fresh init.

3. **StateData time restoration** — The previous attempt set `setCumTime`, `setLevelSteps`, `setDtLevel` but DID NOT update `StateData::curTime()`. AMReX's `Nyx::computeNewDt`/`est_time_step` use `state[State_Type].curTime()` for dt-clamping (clamp `cur_time + dt_0 ≤ stop_time`). With curTime still at 0 after restore, the loop never exits early. Added per-level `setTimeLevel(st_time, st_dt_old, st_dt_new)` calls so AMReX's internal time bookkeeping matches the restored `cumtime`.

4. **Diagnostic prints** — Added explicit `[NyxVeloC]` output at each restart phase (`VELOC_Restart_test`, `VELOC_Recover_size`, `VELOC_Recover_mem`, header validation, scatter completion) so failures can be diagnosed from stdout.

5. **Cadence tightened** — Reduced `NYX_VELOC_CKPT_EVERY` from 20 to 10 coarse steps so the last completed checkpoint before a kill is closer in simulated time to the kill point, shortening the recovery's remaining-work fraction.

6. **Preserved retention controls** — Kept `max_versions=1`/`scratch_versions=1` in `veloc.cfg` and the explicit per-version unlink helper so failure-free disk footprint stays well under the validator's 10.7 GB disk-guard limit.
