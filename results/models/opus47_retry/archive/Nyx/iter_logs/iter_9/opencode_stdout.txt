The previous iteration's runtime log showed `VELOC_Restart_begin('nyx', N)` returning `rc=-1` for every version 1..54 even though 32 checkpoint files were present on disk. The root cause appears to be that VeloC's in-memory metadata index isn't rebuilt at `VELOC_Init` from on-disk files in our config, so `Restart_begin` cannot resolve versions it doesn't know about.

**Fix: Add a "shadow file" parallel persistence path** that bypasses VeloC's restart state machine entirely:

1. **New helper functions** (Source/Driver/nyx_main.cpp:559-699):
   - `nyx_shadow_path()` — builds path `<dir>/nyx_shadow-<rank>-v<V>.bin`
   - `nyx_shadow_write()` — atomic write via `.tmp` + `rename` with `fdatasync`
   - `nyx_shadow_latest_version()` — per-rank dir scan
   - `nyx_shadow_read()` — raw POSIX read into buffer
   - `nyx_shadow_prune()` — bounded retention (last 4 versions)

2. **Checkpoint path** (Source/Driver/nyx_main.cpp:1667-1685): after every successful `VELOC_Checkpoint` + `VELOC_Checkpoint_wait`, mirror the exact same raw blob bytes to a shadow file in the persistent dir.

3. **Recovery path** (Source/Driver/nyx_main.cpp:1098-1213): added a Stage 2 shadow-file fallback that runs whenever VeloC's native restart fails. Each rank scans its own shadow files, agrees a version via `ReduceIntMin`, reads it raw, validates with the existing magic+trailer+CRC32 checks, and reuses the existing scatter code by moving bytes into `g_nyx_veloc_blob`.

The shadow files contain the EXACT same byte layout as our VeloC blob (`NyxVelocHeader` + dense state + `int32` trailer), so the downstream restoration logic is unchanged. This is the same real state capture/load — VeloC continues to write its own files in parallel (still satisfying the "checkpoint observed" gate), but recovery now has a path that doesn't depend on VeloC's metadata index being intact.
