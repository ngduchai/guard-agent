What changed:

- `src/update.cpp`
  - `veloc_restart_if_available()` now retries older checkpoint versions if the newest reported VeloC checkpoint cannot be recovered.
  - This addresses the validator symptom where checkpoint files existed and correctness passed, but recovery replayed almost the full run (`recovery_actually_resumed=False`, `recovery/Z_P=1.004`).
  - Particle data is now recovered into a temporary heap buffer first and only copied into `particle->particles` after all ranks agree recovery succeeded.
  - Particle checkpoint size validation now uses `bigint` for expected byte count to avoid integer overflow.

- `src/run.cpp`
  - Preserves the original target timestep across restart.
  - If recovery restores `ntimestep` to a checkpointed value, the resumed run now executes only `target_step - ntimestep` steps instead of extending `laststep` by the restored timestep.

- `veloc.cfg`
  - Verified present at source root and validator-compatible:
    - `scratch = /tmp/sparta_veloc_scratch`
    - `persistent = /tmp/sparta_veloc_persistent`
    - `mode = sync`

Prior rejection diagnosis:

- Checkpoints were observed.
- Failure-prone and failure-free outputs were correct.
- Validation failed because recovery did not actually resume:
  - `recovery_actually_resumed=False`
  - `recovery/Z_P=1.004`
  - total ratio was `1.62x`, above the app’s stricter `1.2x` cap.
- Root cause was likely fallback to a near-full rerun after restart failed or after run bounds were extended incorrectly.
