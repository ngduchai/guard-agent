Implemented fixes targeting the validator’s immediate startup collapse and preserving real VeloC checkpoint/restart behavior.

Changed:

- `models/phold/phold.main.c`
  - Uses the VeloC header by absolute path so PHOLD can call libveloc without adding `/home/ndhai/usr/include` globally.
  - Switches initialization from collective `VELOC_Init(MPI_COMM_ROSS, ...)` to standalone `VELOC_Init_single(g_tw_mynode, ...)`, using the MPI rank as the VeloC process id.
  - Keeps real checkpoint buffers registered with `VELOC_Mem_protect`.
  - Gates the GVT checkpoint hook installation on successful VeloC initialization.
  - Adds a guard inside the GVT hook so failed VeloC init cannot mutate the simulation by rolling back/canceling events without saving.
  - Preserves the existing checkpoint payload: LP state, RNG streams, core RNG streams, queued event metadata, event messages, GVT, GVT count, and PE sequence number.

- `models/phold/CMakeLists.txt`
  - Removed global `INCLUDE_DIRECTORIES(/home/ndhai/usr/include)` to avoid compiling through the VeloC-provided `mpi.h` trampoline.
  - Keeps linking against `veloc-client` and `veloc-modules`.

Confirmed:

- `veloc.cfg` already exists at the source root with validator-compatible absolute `/tmp` paths:
  - `scratch = /tmp/ross_veloc_scratch`
  - `persistent = /tmp/ross_veloc_persistent`
  - `mode = sync`

Why this addresses the rejection:

- The previous validator run showed the resilient binary exited with MPI abort code `1` almost immediately and produced zero checkpoint files.
- The prior implementation had VeloC include/link setup that could shadow MPI headers via `/home/ndhai/usr/include/mpi.h`, and it installed the GVT hook even if VeloC initialization failed.
- The updated code keeps VeloC integration local to PHOLD, avoids collective VeloC startup as a source of early MPI abort, and only activates checkpoint scheduling after successful VeloC initialization.
