The quiesce helper as first written only waited for completed sends, which is not strong enough for a clean snapshot if matching receives complete slightly later. I’m tightening it to keep polling network progress collectively after sends drain before taking the snapshot.
The validator uses `--synch=3`, which is the optimistic scheduler. That means a correct checkpoint cannot just save the priority queue; it also must avoid capturing transient rollback/cancel state. I’m checking the fossil collection behavior around GVT to decide whether snapshotting at GVT is sufficient after quiescing.
There is already an internal ROSS helper intended exactly for checkpoint hooks: it rolls back LPs to GVT and drains cancel/event queues. I’m going to use that before saving, instead of only quiescing MPI, so the snapshot captures committed state plus future PQ events.
Calling the rollback helper likely changes GVT bookkeeping during checkpoint. I’m keeping saved GVT metadata after that call, which is what restart should resume from. Final static checks now.
Implemented the recovery fixes for the iteration 10 timeout/deadlock failure.

Changed files:

- `models/phold/phold.main.c`
- `core/network-mpi.c`
- `core/network-mpi.h`

What changed:

- Recovery now selects a checkpoint version collectively across all ranks using the minimum valid version available on every rank. This avoids ranks restarting from different checkpoint versions after a kill during checkpoint publication.
- Checkpoint metadata now persists full GVT signature state under `USE_RAND_TIEBREAKER`, not just `recv_ts`.
- Restore now restores full GVT/GVT-prev state and KP last-time/signature consistently.
- Checkpoints are allowed after recovery instead of being suppressed, so recovery progress can continue normally.
- Checkpoint validation no longer rejects otherwise valid checkpoints just because the saved event queue exceeds the initial buffer guess; restore resizes as needed.
- Before checkpointing, PHOLD now uses ROSS’s rollback/cancel cleanup helper to snapshot committed GVT state plus future queue events, rather than speculative optimistic state.
- Added `tw_net_quiesce()` to drain MPI send/receive progress before snapshotting, so remote in-flight events are captured into normal queues before persistence.
- Confirmed `veloc.cfg` is already present at source root with absolute `/tmp` scratch/persistent paths.

Why this targets the validator failure:

- Iteration 10 showed checkpoints were observed and failure-free output passed, but recovery timed out with no `validation_output.bin`.
- The likely causes were inconsistent per-rank restart version selection and incomplete optimistic/MPI state capture at checkpoint time.
- The new logic makes restart version selection rank-consistent and snapshots a clean GVT boundary after draining rollback/cancel/network state.
